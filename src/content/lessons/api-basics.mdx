Think about how humans communicate first. People speak many different languages. When people from different countries need to work together, they often use English as a shared language. They do not have to use English for every conversation; it is simply a common language that gives both sides a reasonable chance of understanding one another.

Software programs face a similar problem. A shopping app may need a payment service, a map app may need route information, and a browser page may need data from a server. These programs may have been created by different teams using different technologies, so they need a shared way to communicate.

<img src={`${import.meta.env.BASE_URL}/images/api-shared-language.svg`} alt="Diagram comparing people who use a shared language for international communication with software programs that use APIs and JSON to communicate." />

<aside class="model-purpose">
<p><strong>How can two different software programs ask each other for information or actions?</strong></p>
<p>One program cannot simply look inside another program. They need a clear, agreed way to send requests and receive answers.</p>
</aside>

A program written by one team cannot assume that another program will understand its private words or internal code. The two programs need a standard way to ask questions, describe answers, and report problems.

An <strong>API</strong> provides that agreed conversation. API stands for **Application Programming Interface**, but you do not need to memorize the long name yet. In this lesson, think of an API as the rules for how one program asks another program for information or an action, and how the other program sends its answer back.

<strong>JSON</strong> is one popular format used to carry information in that conversation. It is like the shared language used for the message itself: it gives both programs a predictable way to label and read the data.


## What is JSON?

Before looking at a real response, understand the format many APIs use to answer.

The API needs to send its answer in a form that both people and programs can understand. <strong>JSON</strong> is a very common way to write organized information as plain text.

### JSON objects and lists

Most of the JSON you see will use two shapes:

<table>
<thead><tr><th>Type</th><th>What it looks like in JSON</th><th>What it means</th></tr></thead>
<tbody>
<tr><td><strong>Object</strong></td><td><code>&#123; "name": "Call Taylor", "done": false &#125;</code></td><td>Objects have keys and values. Here, there are 2 keys, "name" and "done". The "name" key has the <em>Call Taylor</em> value. The "done" key has the value <em>false</em>.</td></tr>
<tr><td><strong>Array</strong></td><td><code>[ "Water plants", "Feed cat" ]</code></td><td>A list containing two text values.</td></tr>
</tbody>
</table>

The curly braces <code>&#123; &#125;</code> surround an <strong>object</strong>. An object is a group of named fields that describe one thing. The square brackets <code>[ ]</code> surround an <strong>array</strong>. An array is an ordered list of values.

For example, this JSON object describes one cat. The <code>tags</code> field contains a list inside that object:

```json
{
  "id": "mcje1vkz08SlYsHK",
  "tags": ["cute", "sleepy"],
  "age": 8
}
```

You can map the object’s labelled fields into a table:

<table>
<thead><tr><th>JSON label</th><th>Value from the response</th></tr></thead>
<tbody>
<tr><td><code>id</code></td><td>mcje1vkz08SlYsHK</td></tr>
<tr><td><code>age</code></td><td>8</td></tr>
<tr><td><code>tags</code></td><td><ul class=""><li>cute</li><li>sleepy</li></ul></td></tr>
</tbody>
</table>


<aside class="model-purpose">
<p><strong>Why is JSON so popular?</strong></p>
<p>JSON is small, readable, and supported by almost every modern programming language. A browser, server, phone app, and many different services can exchange the same JSON response without needing to use the same programming language or technology.</p>
<p>Other formats exist, and some are useful for particular jobs. JSON became common for web APIs because it is simple enough for a person to inspect and predictable enough for a program to read. That makes it a practical shared language between different programs.</p>
</aside>


## Example: asking for the visitor's public IP address

We will use a small service at:

```text
https://api64.ipify.org?format=json
```

This service answers the question: “Which public internet address is my browser using to reach you?” In other words, it reports the address that the outside internet sees for the network connection. That is usually different from a private address assigned only to your individual device.

IPify returns a JSON object with an <code>ip</code> label. The value is not fixed: it depends on the public address that makes the request.

The stable shape of the response is:

```json
{
  "ip": "the public address returned for this request"
}
```

The label <code>ip</code> is part of the agreed response shape. The value beside it changes depending on who makes the request.

This page is using the API right now. When your browser loaded the page, its JavaScript asked IPify for the response and placed the actual value returned for your request into the box:

<IpifyLive />

The value is the public address that the API sees. It may belong to your home network, school, workplace, VPN, or another network service. It is not necessarily the private address of your individual device, and it does not reveal your exact physical location.

## From an IP address to a broad location

An IP address is not usually meaningful to a person on its own. A page might use a second API to ask for broad location information associated with that public address.

We will use [FreeIPAPI](https://freeipapi.com/docs/api-reference/get-ip-info), an HTTPS API that can look up broad information about an IP address:

```text
https://free.freeipapi.com/api/json/{ip-address}
```

The part written as `{ip-address}` is replaced with the IP value returned by IPify. FreeIPAPI also allows the IP to be omitted and will use the address making the request, but passing the IP explicitly makes the two-request story easier to see.

A successful response may look like:

```json
{
  "ipVersion": 4,
  "ipAddress": "203.0.113.42",
  "countryName": "United States",
  "regionName": "California",
  "cityName": "Mountain View",
  "isProxy": false
}
```

The exact result depends on the network, and IP geolocation is approximate. It may identify a nearby city, an internet provider, a VPN, a school, or a company rather than the person's exact location. It should not be presented as a precise address or proof of where a person physically is.

### How the two APIs work together

The browser can use the result from one API to ask another API a new question:

<ol class="mental-model-steps">
<li>Ask IPify for the public IP address.</li>
<li>Read the <code>ip</code> value from the JSON response.</li>
<li>Ask FreeIPAPI for broad location information about that IP address.</li>
<li>Read the country, region, and city fields from the second JSON response.</li>
<li>Display the result with honest wording such as “approximate network location.”</li>
</ol>

This is called chaining requests: the answer from one program becomes part of the next request.

This second component performs that chained example live. It first asks IPify for the visitor's public IP, then asks FreeIPAPI for broad location information about that IP:

## A second example: asking an API for a cat

The IPify example returns one small value: an IP address. APIs can return many other kinds of information.

Open this endpoint:

```text
https://cataas.com/cat?json=true
```

It returns information about a cat and includes a URL where the cat image can be loaded. A response might look like this:

```json
{
  "id": "mcje1vkz08SlYsHK",
  "tags": ["cute"],
  "created_at": "2024-09-15T15:50:48.649Z",
  "url": "https://cataas.com/cat/mcje1vkz08SlYsHK?position=center",
  "mimetype": "image/png"
}
```

The exact cat and exact values may change when you ask again. Read the response as a labelled description:

## Build the CATAAS page yourself

Now create a second file called `cat.html`. Your page should ask CATAAS for JSON and turn the response into a small cat card.

Before writing the code, predict:

<ul class="prediction-list">
<li>What should appear while the request is waiting?</li>
<li>Which JSON labels will become visible text?</li>
<li>Which label will tell the page where to find the image?</li>
<li>What should the page show if the request fails?</li>
</ul>

Use this complete example first:

```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>A cat from an API</title>
    <style>
      body {
        max-width: 38rem;
        margin: 2rem auto;
        padding: 0 1rem;
        font-family: system-ui, sans-serif;
      }

      img {
        display: block;
        width: 100%;
        max-height: 28rem;
        object-fit: contain;
        margin-top: 1rem;
      }

      li {
        margin: .4rem 0;
      }
    </style>
  </head>
  <body>
    <main>
      <h1>A cat from an API</h1>
      <p id="status">Asking CATAAS for a cat…</p>

      <section id="cat-card" hidden>
        <img id="cat-image" alt="A cat returned by CATAAS">
        <ul>
          <li>ID: <span id="cat-id"></span></li>
          <li>Tags: <span id="cat-tags"></span></li>
          <li>Created: <span id="cat-created"></span></li>
          <li>File type: <span id="cat-type"></span></li>
        </ul>
      </section>
    </main>

    <script>
      const status = document.querySelector('#status');
      const card = document.querySelector('#cat-card');
      const image = document.querySelector('#cat-image');
      const id = document.querySelector('#cat-id');
      const tags = document.querySelector('#cat-tags');
      const created = document.querySelector('#cat-created');
      const type = document.querySelector('#cat-type');

      fetch('https://cataas.com/cat?json=true')
        .then((response) => {
          if (!response.ok) {
            throw new Error(`Request failed with status ${response.status}`);
          }
          return response.json();
        })
        .then((cat) => {
          if (!cat.url || !cat.id) {
            throw new Error('The response was missing the cat details we need');
          }

          image.src = cat.url;
          id.textContent = cat.id;
          tags.textContent = cat.tags?.join(', ') || 'No tags';
          created.textContent = cat.created_at || 'Unknown';
          type.textContent = cat.mimetype || 'Unknown';
          card.hidden = false;
          status.textContent = 'The API answered. Here is the cat information:';
        })
        .catch((error) => {
          status.textContent = `The cat could not be loaded: ${error.message}`;
        });
    </script>
  </body>
</html>
```

The key parsing step is `response.json()`. It turns the response text into a JavaScript value so your page can read labels such as `cat.id`, `cat.tags`, and `cat.url`.

Check your page in the browser and Network panel:

<ol class="mental-model-steps">
<li>The page shows a waiting message.</li>
<li>The browser requests `https://cataas.com/cat?json=true`.</li>
<li>The response is parsed as JSON.</li>
<li>The cat image appears using the returned `url`.</li>
<li>The returned ID, tags, creation time, and file type appear as HTML text.</li>
<li>A failed request produces an honest error instead of an empty or invented cat card.</li>
</ol>

## The request and response story

When this page asks IPify for the address, the interaction is:

<ol class="mental-model-steps">

<li>The request travels from your browser to the IPify service.</li>
<li>IPify works out which public address made the request.</li>
<li>IPify sends back a response containing JSON data.</li>
<li>The page reads the <code>ip</code> value.</li>
<li>The page changes the screen to display that value.</li>

## Build your own tiny HTML page

Create a new folder with a file named:

```text
index.html
```

Use this complete example first. Read it before opening it in the browser:

```html
<!doctype html>
<html lang="en">
  <head>
    <meta charset="utf-8">
    <meta name="viewport" content="width=device-width, initial-scale=1">
    <title>My public IP address</title>
  </head>
  <body>
    <main>
      <h1>Your public IP address</h1>
      <p id="status">Asking the API…</p>
      <p id="address">Waiting for an answer</p>
    </main>

    <script>
      const status = document.querySelector('#status');
      const address = document.querySelector('#address');

      fetch('https://api64.ipify.org?format=json')
        .then((response) => {
          if (!response.ok) {
            throw new Error(`Request failed with status ${response.status}`);
          }
          return response.json();
        })
        .then((data) => {
          if (!data.ip) {
            throw new Error('The response did not contain an IP address');
          }

          address.textContent = data.ip;
          status.textContent = 'The API answered.';
        })
        .catch((error) => {
          address.textContent = 'The address could not be loaded.';
          status.textContent = error.message;
        });
    </script>
  </body>
</html>
```

The important parts are:

<table>
<thead><tr><th>Code</th><th>Job</th></tr></thead>
<tbody>
<tr><td><code>fetch(...)</code></td><td>Asks the endpoint for a response.</td></tr>
<tr><td><code>response.ok</code></td><td>Checks whether the response reported success.</td></tr>
<tr><td><code>response.json()</code></td><td>Reads the response as organized JSON data.</td></tr>
<tr><td><code>data.ip</code></td><td>Reads the IP value from the response.</td></tr>
<tr><td><code>textContent</code></td><td>Changes what the user sees on the page.</td></tr>
<tr><td><code>catch</code></td><td>Shows honest feedback if the request or response cannot be used.</td></tr>
</tbody>
</table>

## What this example teaches

This tiny page demonstrates a pattern used by much larger applications:

<ol class="mental-model-steps">
<li>A user needs information.</li>
<li>The browser asks another program through an API.</li>
<li>The other program checks or finds the information.</li>
<li>The browser waits for the answer.</li>
<li>The browser displays success or failure honestly.</li>
</ol>

The same pattern can ask for a map route, search messages, load a profile, check an order, or submit a form. The difference is what the API does with the request and what rules it applies.

It also gives you a debugging map. If the address is missing, ask:

<ul class="question-list">
<li>Did the browser send the request?</li>
<li>Did the request use the right endpoint?</li>
<li>Did the API return a successful response?</li>
<li>Did the response contain the field the page expected?</li>
<li>Did the JavaScript read the field correctly?</li>
<li>Did the screen display the value correctly?</li>
</ul>
