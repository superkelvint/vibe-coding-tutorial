After installing, open a terminal and check Git with:

```bash
git --version
```

You should see a version number. You do not need to understand the version number yet; it simply confirms that the command is available.

Git is a tool that keeps a history of the changes you make to a project.
We will use it to create clear local save points after work that we have checked.
You do not need an online account for this lesson; everything happens on your own computer.

Keep this nearby while you practice: <a href={`${import.meta.env.BASE_URL}/lessons/preamble/05-preamble-git-cheatsheet/`}>Git cheat sheet: 10 commands to know</a>.

This hour stays local on purpose. GitHub is a separate online service, not part of Git itself. When you later want to upload a checked project or connect it to Vercel, use the standalone <a href={`${import.meta.env.BASE_URL}/lessons/git-and-github-guide/`}>Git and GitHub guide</a>.

Create a folder called:

```text
vibe-prework
```

Inside it, create:

```text
notes.txt
```

Write this exact line:

```text
My first Git save point.
```

Run each command one at a time:

```bash
git init
git status
git add notes.txt
git commit -m "Add first notes file"
git status
```

### Pause and predict

Before each command, ask:

```text
What do you think this command will change?

Does it change the file, Git's memory, or neither?
```

### Verify the result

Check these separately:

1. Open `notes.txt` and confirm the sentence is present.
2. Open the folder in the browser only if you have a suitable local file view; the browser is not Git and does not create commits.
3. Read the final `git status` and confirm it reports a clean working tree.

<Checkpoint id="preamble-hour1-git-check">I ran the Git commands and confirmed that the final status shows the work is saved and clean.</Checkpoint>

### Make a second change

Git becomes more useful when there is more than one save point to compare. Add this second line to `notes.txt`:

```text
I can compare my changes.
```

Before saving it, predict what Git should notice. Then run:

```bash
git status
git diff
```

The `git diff` output should show the new line as a change that is not saved yet. The file is still ordinary text; Git is showing you the difference between the last commit and the file as it looks now.

Now create the second save point:

```bash
git add notes.txt
git diff --staged
git commit -m "Add comparison note"
git status
```

`git diff --staged` lets you inspect what will be included before committing. The final `git status` should again show a clean working tree.

<Checkpoint id="preamble-hour1-second-change">I added the second line, used `git diff` to see the unsaved change, and confirmed that the file contained the line I intended.</Checkpoint>

### View the history and compare save points

Now ask Git to show the two save points:

```bash
git log --oneline
```

You should see two commits, with the newest one first. The messages are the descriptions you wrote when you created the save points.

To compare the newest commit with the one immediately before it, run:

```bash
git diff HEAD~1 HEAD
```

This should show the second line that was added. In plain English, you just asked Git: “What changed between my first save point and my second save point?”

<Response id="preamble-hour1-history-prediction" prompt="Before running git log --oneline and git diff HEAD~1 HEAD, predict how many save points you will see and what line the comparison should show." rows={8} />

<Checkpoint id="preamble-hour1-history-check">I viewed the two commits in the history and compared them, confirming that the difference matched my second change.</Checkpoint>

### Explain back

You should be able to say:

```text
The file is the work.

Git noticed the file.

I chose the file for the save point.

The commit created the save point.

The history shows my save points.

The diff shows how one save point differs from another.
```

Open `notes.txt` again after the second commit. Both lines should still be present, and the two-commit history should still be visible. If the file changed unexpectedly, inspect the diff before continuing.
