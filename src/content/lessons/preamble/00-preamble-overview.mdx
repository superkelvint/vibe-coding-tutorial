Welcome. The next four hours are preparation for the seven-day build week.

This is not a test of whether you already know how to code. It is a gentle introduction to the few ideas you will need before we start building a real browser application with Codex.

## What the preamble is for

The preamble gives you a small, understandable foundation:

<ul class="observation-checklist">
<li>where a project lives;</li>
<li>what files are;</li>
<li>how a browser reads a webpage;</li>
<li>what HTML and CSS do;</li>
<li>how JavaScript notices a click and changes the page;</li>
<li>how Git records a save point; and</li>
<li>how to investigate a user problem before asking an agent to build a solution.</li>
</ul>

You will make very small things on purpose. Small examples are easier to inspect, easier to break, and easier to explain. That gives you a reliable mental map before the course introduces React, servers, SQLite, shared data, and deployment.

<aside class="mantra-callout">
<p>You are not here to create software.</p>
<p>You are here to solve user problems.</p>
<p>The preamble gives you enough technical footing to start solving those problems with care.</p>
</aside>

## The four-hour overview

<table>
<thead><tr><th>Hour</th><th>What you will do</th><th>What you should understand afterward</th></tr></thead>
<tbody>
<tr><td>1. Setup, files, browser, and Git</td><td>Create a project folder and file, open it in a browser, and make a Git save point.</td><td>A project is a folder of files; the browser reads some of them; Git records understandable checkpoints.</td></tr>
<tr><td>2. Make a tiny webpage</td><td>Create a small page with a heading, paragraph, list, button, and simple styling.</td><td>HTML says what exists; CSS says how it looks; the browser draws the result.</td></tr>
<tr><td>3. Make a button do something</td><td>Connect one click to one visible change with plain JavaScript.</td><td>The user acts; JavaScript notices; JavaScript changes the page.</td></tr>
<tr><td>4. Product judgment and better prompts</td><td>Talk through a real responsibility problem and turn what you learn into a small, testable request.</td><td>Users often describe solutions; builders investigate the underlying problem before building.</td></tr>
</tbody>
</table>

## What you should be able to do by the end

By the end of the four hours, you should be able to:

<ol class="mental-model-steps">
<li>Find the project folder and the files inside it.</li>
<li>Open a webpage and describe what the browser is reading.</li>
<li>Make a small HTML or CSS change and predict what should change on screen.</li>
<li>Connect one button click to one visible JavaScript change.</li>
<li>Use Git to create a clear save point.</li>
<li>Describe a user’s situation, current workaround, pain, and smallest useful thing to try.</li>
<li>Give Codex a focused prompt with observable success criteria and a check.</li>
</ol>

## What the preamble deliberately does not cover

You do not need to learn these topics yet:

<ul class="example-list">
<li>React;</li>
<li>servers;</li>
<li>databases or SQLite;</li>
<li>accounts and login;</li>
<li>deployment;</li>
<li>advanced JavaScript;</li>
<li>framework configuration; or</li>
<li>large application architecture.</li>
</ul>

Those topics arrive later, after the browser and product foundations are in place. The goal is not to memorize a giant technology list. The goal is to understand one small layer at a time.

## How to use the four hours

For each hour:

<ol class="mental-model-steps">
<li>Read the mental model in plain English.</li>
<li>Predict what should happen before running a command or opening the browser.</li>
<li>Ask Codex for a small, explained change.</li>
<li>Look at the actual result.</li>
<li>Try one safe edge case or deliberate failure.</li>
<li>Record what passed, what failed, and what remains unclear.</li>
<li>Explain the idea back before moving on.</li>
</ol>
