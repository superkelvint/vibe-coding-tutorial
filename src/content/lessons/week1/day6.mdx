Yesterday, the Family Chore Board worked in one browser. Today we will face the problem that browser-only code cannot solve on its own.

## What if two family members use the app?

Imagine this:

<ol class="mental-model-steps">
<li>One parent opens the Chore Board on a laptop.</li>
<li>They add “Take out the trash” and assign it to Maya.</li>
<li>Maya opens the app on her phone.</li>
<li>She needs to see the same chore.</li>
<li>She marks it done.</li>
<li>The parent needs to see that change too.</li>
</ol>

If each browser keeps its own temporary list, the family does not have one Chore Board. They have several unrelated copies that can disagree.

<aside class="model-purpose">
<p><strong>Where should the shared chore list live so that every browser can ask for the same current information?</strong></p>
</aside>

The answer is not “make React remember harder.” React manages the current screen inside one browser. We need another place that all of the browsers can contact.

## The server becomes the shared source of truth

A <strong>server</strong> is another computer or hosted application that receives requests from browsers and sends responses back. It can apply rules, read shared saved data, change that data, and tell the browser what happened.

A <strong>database</strong> is a place for organized saved data that can be found again later. A database is made up of <strong>tables</strong>, and a table is very much like an Excel spreadsheet: it has rows and columns.

Each column has a name that describes what belongs there. Each row is one saved item. In our `chores` table, one row will represent one chore:

<table>
<thead><tr><th><code>id</code></th><th><code>title</code></th><th><code>assigned_to</code></th><th><code>done</code></th><th><code>created_at</code></th></tr></thead>
<tbody>
<tr><td>42</td><td>Take out the trash</td><td>Maya</td><td>false</td><td>Today</td></tr>
<tr><td>43</td><td>Feed the dog</td><td>Leo</td><td>true</td><td>Today</td></tr>
</tbody>
</table>

The column names are not decoration. They tell the server what each value means. Without named columns, the app would not know whether “Maya” was the person, the chore title, or something else. When the app asks for chores, the server reads rows from this table and sends their named values back to the browser.

The server and database work together:

<ol class="mental-model-steps">
<li>The browser asks the server for the current chores.</li>
<li>The server reads the shared saved data.</li>
<li>The server sends the chores back.</li>
<li>The browser displays the response.</li>
<li>When someone adds or completes a chore, the browser sends a request.</li>
<li>The server checks the request and changes the shared saved data.</li>
<li>The server sends a response that says whether the change succeeded.</li>
<li>The browser updates its state and React updates the screen.</li>
</ol>

## Who is responsible for which part?

There are three different responsibilities here. Keeping them separate makes the architecture much easier to understand:

<table>
<thead><tr><th>Part</th><th>Who provides it?</th><th>What we do</th></tr></thead>
<tbody>
<tr><td>Client</td><td>Us, using React in the browser.</td><td>We write the screen, buttons, loading feedback, requests, and response handling that the user sees.</td></tr>
<tr><td>Server</td><td>Us, using server-side JavaScript.</td><td>We write the request handlers, rules, validation, database queries, and responses.</td></tr>
<tr><td>Database engine</td><td>SQLite, an existing open-source program.</td><td>We use SQLite to save and retrieve rows. We do not write SQLite itself.</td></tr>
<tr><td>Database table</td><td>Codex translates the approved product context into a table design.</td><td>We review whether the generated table represents the app correctly; Codex writes the SQL.</td></tr>
</tbody>
</table>

This is similar to using a spreadsheet program. You do not code Excel in order to use a spreadsheet. You describe what information the app needs, and the tool creates or stores the rows and columns. In the same way, we install and use the open-source SQLite engine, while coding the client and server that give the data meaning for the Family Chore Board.

## Walk through the same idea with Instagram

The Chore Board is easier to understand when you connect it to an app you already know. Imagine an Instagram-like app where someone taps the heart on a post.

The browser may show the post, the heart, and the current like count. But the browser is not the shared authority for the like. If it were, every browser could invent its own count.

<table>
<thead><tr><th>Part</th><th>Instagram-like example</th></tr></thead>
<tbody>
<tr><td>Browser</td><td>Shows the post, the heart button, the current count, and “Saving…” feedback.</td></tr>
<tr><td>Request</td><td>The browser asks to create or remove this user’s like on this post.</td></tr>
<tr><td>Server rule</td><td>The server checks that the request is allowed and that the same user cannot create five likes by accident.</td></tr>
<tr><td>Shared saved data</td><td>Stores rows describing the post, the user’s like, and the count or the information needed to calculate it.</td></tr>
<tr><td>Response</td><td>Says whether the like was saved or whether the action failed.</td></tr>
<tr><td>Screen update</td><td>React shows the confirmed heart state, count, and feedback returned by the app.</td></tr>
</tbody>
</table>

The complete interaction looks like this:

<ol class="mental-model-steps">
<li>The browser loads a post and asks for its current like information.</li>
<li>The server reads the post and the saved like data.</li>
<li>The server sends the post, like state, and count back.</li>
<li>The browser displays the heart and count.</li>
<li>The user taps the heart.</li>
<li>The browser shows waiting feedback and sends a request.</li>
<li>The server checks the request and saves or removes one like.</li>
<li>The server sends a success or failure response.</li>
<li>The browser updates its state and React redraws the heart, count, and feedback.</li>
</ol>

The same failure questions apply to the Chore Board:

<ul class="question-list">
<li>Did the browser send the request?</li>
<li>Did the server accept it?</li>
<li>Was the like actually saved?</li>
<li>Did the response describe what happened accurately?</li>
<li>Did the browser redraw from the response, or merely pretend the action worked?</li>
</ul>

This is why the architecture is useful. It does not just tell you which technologies exist. It gives you places to look when the heart looks selected but a refresh shows that the like was never saved.

The app developer is responsible for both sides of this relationship:

<table>
<thead><tr><th>Part</th><th>What it must do well</th></tr></thead>
<tbody>
<tr><td>Client</td><td>Be clear and friendly for the person using it. Show waiting, success, and failure honestly.</td></tr>
<tr><td>Server and saved data</td><td>Be dependable. Check requests carefully, store the right values, and return accurate results.</td></tr>
</tbody>
</table>

The client should not pretend that a change succeeded before it knows. A visible chore is not proof that the shared data was changed.

## Producers and consumers: who decides what is true?

Think of a shop’s inventory. The shop’s inventory system determines how many items really exist. A customer-facing page consumes that information and may let a customer request a purchase, but the page cannot simply declare that the shop has ten more items.

In the Chore Board:

<table>
<thead><tr><th>Role</th><th>In plain English</th></tr></thead>
<tbody>
<tr><td>Producer of shared data</td><td>The server and saved data decide what chores are actually recorded.</td></tr>
<tr><td>Consumer of shared data</td><td>Each browser requests that data and displays the answer.</td></tr>
<tr><td>Allowed change</td><td>A browser may ask to add, complete, or delete a chore, but the server must check and accept the request before shared data changes.</td></tr>
</tbody>
</table>

This is why the server must be the source of truth. If every browser could decide its own version was correct, two people could complete different versions of the same chore and neither screen would reliably represent the family’s shared list.

## Safety for a teaching project

This is a disposable teaching database. Do not place private family information in it.

Never put a privileged server or database secret in browser code. The browser should call the server; it should never open the SQLite file directly. This is a disposable prototype, so treat its data as temporary and disposable.

<details>
<summary>Why this matters</summary>

The browser is visible to anyone who uses the app. A secret that gives unrestricted control over shared data must stay on a protected server-side system. Today’s goal is to understand the client–server–saved-data flow, not to build a production security system.
</details>
