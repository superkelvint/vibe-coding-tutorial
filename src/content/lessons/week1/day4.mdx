Yesterday you learned why many web apps need two cooperating parts: a client for the person using the app and a server for shared, trustworthy information.

Today you will build only the client side of a small product: <b>a Family Chore Board</b>. The chores will live temporarily in this browser. That is intentional. We are learning how to model and build the user-facing workflow before we ask another computer to save or share it.


## Start with the family’s problem

Imagine a family where people regularly ask:

<ul class="question-list">
<li>“Who was supposed to take out the rubbish?”</li>
<li>“Did anyone feed the dog?”</li>
<li>“What still needs to be done?”</li>
<li>“I thought you said you had done it.”</li>
</ul>

The problem is not “we need a web app.” The problem is that responsibilities are easy to forget, hard to see, or unclear between family members.

The family might currently use memory, verbal reminders, paper, or a group chat. Those work sometimes, but information can be missed or become out of date.

<div class="app-example">
<p><strong>Smallest useful version:</strong> one person can add a chore, assign it to a family member, see the list, and mark one chore done.</p>
<p><strong>Success signal:</strong> a person can add three chores, assign them, and mark exactly one complete without changing the others.</p>
</div>

## What we are building today

Today’s version is a browser-only prototype:

<ul class="observation-checklist">
<li>Chores exist only in this browser’s temporary memory.</li>
<li>A person can add a chore.</li>
<li>A person can assign the chore to someone.</li>
<li>The list shows the chore and its assigned person.</li>
<li>A person can mark one chore done.</li>
<li>Blank chores are rejected with understandable feedback.</li>
<li>Refreshing the page clears the list.</li>
</ul>

<aside class="model-purpose">
<p><strong>Scope boundary:</strong> this is useful for learning the workflow, but it is not yet a shared family product. A second browser will not see the same chores, and refreshing will remove them.</p>
</aside>

Do not add login, reminders, points, recurring chores, chat, a database, or a server today. Those might become useful later, but they would distract from the browser workflow we are learning now.

### Why unit and end-to-end tests matter

When you are vibe coding, it is easy to make one part of an app better while accidentally breaking something else. Automated tests act like a safety net: they check that the app still behaves the way you expect after every change.

They also **keep the coding agent honest**.

An AI coding agent can write code that looks reasonable and confidently say a task is complete. But code that looks correct is not necessarily code that works. Without tests, the agent may have no reliable way to verify that it actually met your requirements.

Automated tests turn your instructions into checks the agent can run.

These tests are development-time code. The agent creates and runs them while building the app; they are not user-facing features and do not normally run as part of the app’s runtime experience. Their job is to give the agent a repeatable way to check correctness before the work is considered complete.

A **unit test** checks one small piece of behavior in isolation. For example:

> “A chore with a blank name should not be added.”

This gives the agent a precise rule to satisfy. It can run the test, see whether the rule passes, and revise its code when it does not.

An **end-to-end test** checks a complete user journey through the app. For example:

> “A user can add two chores, mark one as done, and see that the other remains unfinished.”

This verifies that the app works from the user’s point of view—not just that the individual pieces of code appear correct.

When you are vibe coding, you usually are not writing the tests yourself. The coding agent writes and runs them. Your job is to make testing part of the assignment.

Do not only prompt the agent to build a feature. Ask it to:

- create tests for the required behavior,
- run those tests,
- fix any failures, and
- consider edge cases that might not be obvious from the main request.

When a new change breaks something that previously worked, it is called a **regression**. Regressions are usually accidental. The agent may fix the Add button and unknowingly break the Done button, or improve the chore list and accidentally allow blank chores again.

Without tests, both you and the agent are relying heavily on appearances and assumptions. The app may load, the code may look clean, and the agent may report success—even though an important workflow is broken.

With tests, “done” has a clearer meaning. The agent can run the checks and provide evidence that the requested behavior works. Tests do not guarantee that an app has no bugs, but they make it harder for the agent to mistake plausible-looking code for a working solution.

Tests give you an early warning. They help you preserve the progress you have already made, understand exactly what broke, and make new changes with more confidence. Instead of repeatedly rechecking the entire app by hand, you can rely on focused tests to confirm that the important parts still work.

Without that safety net, vibe coding can feel like whack-a-mole: you fix Add, then break Done. You repair the list, then blank input starts working again. Tests help you keep your progress, give the agent a way to verify its own work, and turn confident claims into results that can actually be checked.

### How to create tests

You do not need to know how to write these tests. You do need to tell the agent what outcomes matter.

You should also ask the agent to look beyond the happy path. The happy path is the straightforward case where the user does everything as expected. **Edge cases** are the less obvious situations where bugs often hide.

For a chore app, edge cases might include:

* entering only spaces instead of a chore name,
* clicking Add twice,
* using a very long chore name,
* marking an already completed chore as done,
* refreshing the page after making a change, or
* trying the feature when the list is empty.

You will not always think of every edge case yourself. That is fine. **Ask the agent to identify likely edge cases, explain which ones it plans to test, and add tests for the important ones.** You can then review the list and point out anything it missed.

A useful vibe-coding prompt is:

> “Implement this feature, create unit and end-to-end tests for the required behavior, identify and test important edge cases, run the full test suite, and fix any failures before declaring the task complete.”

Always remember this: 
> **If you do not test your application, you are by default asking your users to test your application for you.**

## Think through the product before writing code

Before asking Codex to build the Family Chore Board, workshop the experience with ChatGPT.

This can feel like “wasting time” because you are writing words instead of code. It is not wasted time. Writing down the requirements and the design forces you to think through the application before code makes the decisions harder to change.

When you write the experience out, you have to notice things such as:

<ul class="question-list">
<li>What does the family member need to do first?</li>
<li>What information must be visible at each step?</li>
<li>What happens when the user makes a mistake?</li>
<li>How does the user know that the chore was added or completed?</li>
<li>What should happen when there are no chores yet?</li>
<li>Which choices are product decisions, and which are merely implementation details?</li>
</ul>

This is valuable because **changes are cheapest before any code is written**. Changing a sentence, a user flow, or a rough mockup is quick. Changing a coded interface can require rewriting markup, state logic, styling, tests, and other behavior that has already grown around the first decision.

<p><strong>Why document the idea?</strong> Documentation is not paperwork added after the real work. It is a way to do the real product thinking before implementation. It gives you a chance to discover confusion, missing states, and unnecessary features while they are still cheap to change.</p>
</aside>

### Step 4: ask ChatGPT for an implementation plan

The PRD explains what problem we are solving. The high-level design explains the experience we chose. Next, ask ChatGPT to suggest a safe way to build that experience in small pieces.

The plan should be broken into **milestones**. Each milestone must produce something visible in the app. After every milestone, you should be able to open the page and see the product taking shape—not just trust that a large invisible refactor is complete.

Visible milestones make the work easier to understand and easier to correct. If the experience looks wrong after the first or second milestone, you can change direction while very little code depends on that decision. A plan that hides all visible output until the end makes it much harder to tell where the product went off course.

Each milestone also needs focused tests or checks. The check should answer: **what would prove that this milestone was implemented correctly?**

```text
Read these approved project documents first:
- docs/product-requirements.md
- docs/high-level-design.md

Suggest an implementation plan for the browser-only Family Chore Board.

Break the plan into small milestones. For every milestone, include:
1. the user-visible result after this milestone
2. the smallest implementation change needed
3. the product behavior being added
4. focused tests or checks that verify this milestone
5. a manual check I can perform in the browser
6. what existing behavior must still work
7. what remains intentionally unbuilt

The milestones should show visible progress, for example:
- show the empty Chore Board screen
- let the user enter and display one chore
- let the user choose and display an assignment
```

### Final test review: happy case and edge cases

It is easy to test only the **happy case**: the user enters a normal chore, chooses a person, clicks Add, and sees the expected result.

That is important, but it is not enough. **Edge cases** are the unusual, incomplete, invalid, or difficult situations that can expose a broken rule:

<table>
<thead><tr><th>Happy case</th><th>Possible edge case</th></tr></thead>
<tbody>
<tr><td>The user enters “Feed the dog” and adds it.</td><td>The user submits an empty name or a name containing only spaces.</td></tr>
<tr><td>The user marks one visible chore done.</td><td>The user clicks Done twice or clicks a button after several chores have been added.</td></tr>
<tr><td>The user assigns a normal person.</td><td>No person is selected, or the selected person is missing from the available choices.</td></tr>
<tr><td>The list contains a few normal chores.</td><td>The list is empty, contains duplicate names, or contains a very long chore name.</td></tr>
<tr><td>The screen updates after the data changes.</td><td>The data changes but the screen shows old information, or the screen changes the wrong chore.</td></tr>
</tbody>
</table>

The happy case tells us that the main path works. Edge-case thinking asks whether the rules still protect the user when the input, timing, or data is not ideal. This matters because real users do not follow the perfect path every time—and a small mistake can make an app confusing or untrustworthy.

At the end of each milestone, ask Codex to perform this critical test review itself. Codex should inspect the implementation and the approved requirements, map each required behavior to a meaningful test or check, inspect coverage, and think through edge cases that the current tests may not exercise. Your job is to read Codex’s report and evidence—not to reproduce the entire test audit manually.

```text

2. For each behavior, identify the unit test or other automated check that
   verifies it. If the project has no unit-test setup, say that clearly.
3. Inspect the available test coverage and identify important code paths that
   are not covered. Do not merely report the coverage percentage.
4. Decide whether each test is meaningful or could pass for the wrong reason.
5. Separate the happy-case checks from edge-case checks.
6. Think critically through edge cases involving empty input, repeated actions, missing
   choices, multiple chores, long names, wrong item selection, stale screen
   data, and refresh behavior where relevant.
7. Recommend the smallest meaningful tests or manual checks for uncovered
   cases. Do not add a testing framework or external library without approval.
8. Run the available checks and report what passed, failed, or was not run.
9. State what remains unverified and why.
```
