Yesterday you learned to look behind an app’s screen. Today you will zoom in on one part of the story: the browser.

You will build one tiny interaction—a Like button is a good choice—and watch what happens when the browser handles everything locally. The change should feel immediate. Then you will refresh the page and discover an important fact:

> A screen can change without the change being saved for later.

This is a learning task. You are not here to create software. You are here to solve user problems, and today’s problem is a small learning problem: understanding what the browser can do by itself.

Here’s all we’re doing today: one page, one browser, one temporary change, and one piece of feedback. Keeping the exercise this small lets us see exactly what the browser is doing.

## The browser-only idea

The application runs in the user’s browser.

The browser can:

<ul class="example-list">
<li>download HTML, CSS, and JavaScript files;</li>
<li>show a screen;</li>
<li>notice clicks and typing;</li>
<li>keep temporary data in memory;</li>
<li>apply rules locally; and</li>
<li>update the screen immediately.</li>
</ul>

The browser-only loop is:

<ol class="mental-model-steps">
<li>The user sees some data.</li>
<li>The user takes an action.</li>
<li>Browser code notices the action.</li>
<li>Temporary data changes in the browser.</li>
<li>The browser updates the screen.</li>
<li>The user receives feedback.</li>
</ol>

<aside class="model-purpose">
<p><strong>Why this matters:</strong> first learn to recognize the interactions that the browser can handle on its own. That makes the system smaller, easier to understand, and easier to debug.</p>
</aside>

Not every interaction needs another computer. A menu can open, a counter can increase, a tab can become selected, and a temporary note can appear entirely inside this page.

### Ask for a plan before asking for code

Use this planning prompt:

```text
Do not write code yet.

I want to build one tiny browser-only interaction.

The user will [action].
The browser should temporarily remember [data].
The screen should show [feedback].

Explain:
1. what the user sees before the action
2. what data the browser remembers
3. what changes after the action
4. what visible feedback proves it worked
5. what should happen after refresh

Keep the page to one HTML file with simple CSS and JavaScript. Do not add extra tools or extra parts.
```

Before sending it, predict what a good answer should contain. It should describe the browser’s responsibility in plain English and keep the task small.


### Evidence before debugging

If the interaction fails, do not immediately ask Codex to rewrite the file. Record what happened first:

<div class="app-example">

<p><strong>Actual:</strong> what did it remember or show instead?</p>
<p><strong>Steps:</strong> what did you do to reproduce it?</p>
<p><strong>Evidence:</strong> what was visible, and what appeared in the browser Console?</p>
<p><strong>Suspected layer:</strong> event, rule, temporary data, or screen update?</p>
</div>

### Run focused checks

Use the cheapest checks that support the browser-only claim:

1. Read the diff or changed file.
2. Open the page and confirm the initial state.
3. Click once and confirm the state and feedback change.
4. Refresh and confirm the initial state returns.
5. Open the browser Network panel and confirm the click itself did not create a data request.
6. Read the Console for errors.

The initial HTML, CSS, and JavaScript files may have been loaded over a network. That is different from the click handler making a request. Your claim is that the interaction itself was handled locally.

### Break it on purpose

Choose one small browser-only bug. Make one change at a time so you can connect the symptom to the broken layer:

<ul class="example-list">
<li>The data changes, but the screen does not update.</li>
<li>The screen changes, but the temporary value does not change correctly.</li>
<li>The click handler changes the wrong item.</li>
<li>The button’s visible name and the expected event connection do not match.</li>
</ul>

### What is still unverified?

This exercise does not prove that the Like is saved, shared with another browser, or safe for multiple users. It proves a smaller claim: one browser can notice an action, change temporary data, update its screen, and provide feedback without contacting another computer for that click.
