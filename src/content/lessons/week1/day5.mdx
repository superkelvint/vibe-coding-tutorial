Yesterday you designed and built the first Family Chore Board workflow in plain browser code. Today you will reorganize that same browser-only product using a tool designed to help keep a changing screen consistent with changing browser data.

This is not a new product. The technical tool is the subject for today—not a reason to add saving after refresh, sharing across browsers, reminders, or extra features.

## One change can affect several parts of the screen

Imagine the Chore Board has several parts of the screen:

<ul class="question-list">
<li>a form where the user enters a chore;</li>
<li>a person selector;</li>
<li>a list of all chores;</li>
<li>a Done button for each chore; and</li>
<li>feedback showing what happened.</li>
</ul>

When the user marks one chore done, all of the relevant parts should agree immediately. The list should show the chore as done, the correct button should change, the other chores should remain unfinished, and the feedback should describe the same result.

In plain browser code, it is possible to update each part separately. But as the interface grows, it becomes easy for one part to show new data while another part still shows the old data. You can fix the list and accidentally forget the count. You can update a copy of the chore but not the data that the rest of the screen uses. You can click Done on one row and change the wrong row.

<aside class="model-purpose">
<p><strong>How can we organize the client so that one change to the browser data reliably updates every part of the screen that depends on it?</strong></p>
</aside>

<img src={`${import.meta.env.BASE_URL}/images/day5-react-state-sync.png`} alt="A fictional Family Chore Board showing one chore marked done consistently in the summary, chore list, status badge, button, and feedback message." />

*Figure 1. The same `done` value appears in several places. A reliable interface keeps all of those views consistent.*

## The answer: React helps synchronize state and the screen

React is an open-source JavaScript tool for building interactive screens out of reusable components. You describe what the screen should show for the current browser data, and React helps update the right parts when that data changes.

For new React projects, use TypeScript by default. See the standalone <a href={`${import.meta.env.BASE_URL}/lessons/typescript-react-guide/`}>TypeScript for React projects guide</a> for the reason and the Codex prompts that keep the project consistently typed.

Imagine the chore list is browser data. A user adds a chore, and that data changes. The screen should change too.

In this lesson, <strong>state</strong> simply means the current browser data that the screen needs to remember while the page is open.

React helps organize that relationship:

<ol class="mental-model-steps">
<li>The app owns the current chore data.</li>
<li>The app gives the relevant data to smaller interface pieces.</li>
<li>The user acts inside one of those pieces.</li>
<li>The piece reports the action.</li>
<li>The app updates its browser state.</li>
<li>React redraws the parts of the screen that depend on that state.</li>
<li>The user sees feedback that matches the current data.</li>
</ol>

<aside class="model-purpose">
<p><strong>The mental model:</strong> React helps keep the screen synchronized with browser state. It is a way to organize the client side of the app.</p>
</aside>

## React components are reusable pieces of a screen

There is another reason React became popular: it gives you a clear way to split a large interface into smaller, reusable pieces called <strong>components</strong>.

Imagine building the Chore Board as one giant block of code. The form, the list, every chore row, every button, and every message would all be mixed together. That can work for a tiny page, but it becomes harder to understand when the screen grows.

A component is a named piece of the interface with a focused job. It can receive information, display that information, and report an action back to the part of the app that owns the data.

For example, the same `ChoreItem` component can display every row in the list:

<table>
<thead><tr><th>Component</th><th>What it does</th><th>What it needs</th></tr></thead>
<tbody>
<tr><td><code>ChoreForm</code></td><td>Shows the fields for adding a chore.</td><td>The people the user can choose and what should happen after Add.</td></tr>
<tr><td><code>ChoreList</code></td><td>Shows the current collection of chores.</td><td>The current chore data.</td></tr>
<tr><td><code>ChoreItem</code></td><td>Shows one chore and its Done button.</td><td>One chore and a way to report which chore was clicked.</td></tr>
<tr><td><code>Button</code></td><td>Provides a consistent, understandable button.</td><td>Its label, whether it is available, and what should happen when it is clicked.</td></tr>
</tbody>
</table>

The benefit is not “more files.” The benefit is that each piece has a job you can explain, reuse, and check. If the list contains five chores, React can render five `ChoreItem` components from the same pattern while giving each one different chore data.

<img src={`${import.meta.env.BASE_URL}/images/day5-react-choreitem-reuse.png`} alt="A reusable ChoreItem component on the left connected to three chore rows that use the same layout with different data." />

*Figure 2. One `ChoreItem` pattern is reused for three chores. The layout stays consistent while the title, person, status, and action value change.*

### A real open-source example: React DayPicker

[React DayPicker](https://react-day-picker.js.org/) is a popular open-source React date-picker component. It is a good example because a date picker is much more than a text box: it has to display months, let someone move between months, select one date or a range, disable dates that cannot be chosen, and support keyboard navigation.

<img src={`${import.meta.env.BASE_URL}/images/day5-react-daypicker.png`} alt="A fictional travel form using a React date-picker component with range selection, disabled dates, month navigation, and keyboard focus." />

*Figure 3. A date-picker component handles several interaction details that would take much more work than a plain HTML input. The surrounding app still decides what the selected dates mean and what happens next.*

The component handles the calendar interaction, but it does not decide what the selected dates mean. Your application still has to receive the selected value, check whether it makes sense for the product, update its own data, and give the user honest feedback.

Today you will install React DayPicker and use it in a small, separate component demonstration. This gives you a real component-library experience without quietly changing the Family Chore Board’s product behavior. You will see the component on screen, select a date, and verify that your React code receives the selected value.

<aside class="model-purpose">
<p><strong>Component mental model:</strong> a component is a reusable interface piece; <strong>props</strong> are the information it receives about what to display; an <strong>event</strong> is a signal that tells the data owner what the user did; the data owner remains responsible for changing the current data.</p>
</aside>

React does not automatically make the chore list:

<ul class="example-list">
<li>come back after you refresh the page;</li>
<li>appear in another browser;</li>
<li>recognize a particular person; or</li>
<li>gain reminders or other new product behavior.</li>
</ul>

A React application can still be entirely client-side. In today’s version, the chores should still disappear after refresh, just as they did in Day 4.
