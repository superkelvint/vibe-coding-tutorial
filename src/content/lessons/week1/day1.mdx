You already know how to use apps. Today you get to look at them slightly differently.

Pick three apps you actually use—or know well enough to talk about. WhatsApp, Instagram, YouTube, Amazon, Uber, Gmail, Google Maps, online banking, and Facebook are all fine. The specific apps do not matter. Familiarity does.

We are going to follow one simple question:

> When a person does something in an app, what does the app remember, what rules does it check, and how does it show the result?

This is not a coding day. You are not expected to build anything, understand an unfamiliar repository, or memorize framework vocabulary. You are learning to see the machinery behind screens you already use.


## First, know what is—and is not—on the agenda

Today is about getting a mental map that will make the later build feel less mysterious.

By the end of the course, the bigger story will look like this:

<ol class="mental-model-steps">
<li>A user sees a screen.</li>
<li>The user takes an action.</li>
<li>The app notices the action.</li>
<li>The app changes what it remembers.</li>
<li>Rules decide what should happen next.</li>
<li>The screen updates.</li>
<li>Sometimes the app asks another computer for data.</li>
<li>Sometimes that data is saved in a shared database.</li>
<li>Deployment lets another person use the app from a real URL.</li>
</ol>

Today we are starting with the first five pieces:

```text
users → data → actions → rules → feedback
```

## The five ideas hiding inside almost every app

Here is the whole mental model:

<ol class="mental-model-steps">
<li>The app displays some data to the user.</li>
<li>The user takes an action.</li>
<li>The app checks a rule.</li>
<li>The app reads or changes data.</li>
<li>The app gives feedback.</li>
</ol>

<aside class="model-purpose">
<p><strong>Why are we learning this model?</strong></p>
<p>Because it gives us a first mental model for <em>how the app is going to be implemented</em>. Later, when we look at a browser, React, a server, a database, or deployment, we can ask which part of this story we are looking at. We do not need to memorize every technical detail yet—we just need a map for where each detail belongs.</p>
<p>It also gives us a framework to systematically identify the <b>root cause</b> of application bugs. If something looks wrong, we can ask where the story stopped matching reality: did the action happen, did a rule reject it, was the data saved, is the server returning an old result, or is the screen showing stale feedback? That helps us isolate the root cause instead of guessing.</p>
</aside>

A user might be a customer, a friend, a driver, a student, or an employee. Data might be a message, photo, order, location, bank balance, or chore. An action might create, read, change, or delete something. A rule decides whether that action is allowed. Feedback tells the user what happened.

### Temporary versus saved

Two words are useful here:

<dl class="definition-list">
<div><dt><b>Ephemeral data</b></dt><dd>Temporary information used during an interaction.</dd></div>
<div><dt><b>Persistent data</b></dt><dd>Information saved so it can be available later.</dd></div>
</dl>

Examples:

| Example | Type |
| --- | --- |
| WhatsApp message being typed | Ephemeral |
| WhatsApp message after Send succeeds | Persistent |
| Instagram caption before publishing | Ephemeral |
| Published Instagram post | Persistent |
| Amazon filter currently selected | Often ephemeral |
| Completed Amazon order | Persistent |

One important warning: visible does not automatically mean saved. A filter can be visible while you are using it and disappear when you refresh. A sent message can remain visible because the app saved it and can retrieve it later.

<img src={`${import.meta.env.BASE_URL}images/day1-temporary-vs-saved.png`} alt="A fictional shopping app comparison showing a temporary size filter disappearing after refresh and a saved order remaining visible." />

*Figure 2. “Visible now” and “saved for later” are different claims. A refresh is a simple way to learn which claim you are actually seeing.*

Try these examples:

| Example | What to expect |
| --- | --- |
| Typing a comment before posting | Temporary |
| The posted comment | Expected to be saved |
| A selected song in a music queue | Often temporary |
| A playlist you created | Expected to be saved |
| A hotel date picker before you search | Temporary |
| A confirmed reservation | Expected to be saved |
| A map zoom level | Often temporary |
| A saved place or starred address | Expected to be saved |
| A form with an unsent draft | May be temporary—or may be saved as a draft |

<aside class="model-purpose">
<p><strong>The useful question:</strong> What happens after you refresh or return later?</p>
</aside>

### Feedback is not decoration

Good feedback answers questions the user is already asking:

<ul class="question-list">
<li>Did the app notice what I did?</li>
<li>Is it still working?</li>
<li>Did it succeed?</li>
<li>Did it fail?</li>
<li>What changed?</li>
<li>What should I do next?</li>
</ul>

Some familiar examples:

<ul class="example-list">
<li><strong>Instagram:</strong> the heart changes and the count may increase.</li>
<li><strong>WhatsApp:</strong> a message shows sending, sent, delivered, or failed.</li>
<li><strong>Amazon:</strong> the cart count changes, or an error explains what went wrong.</li>
<li><strong>Uber:</strong> the app shows searching, driver assigned, arrival time, or no driver available.</li>
</ul>

Here are some less obvious examples of feedback:

<ul class="example-list">
<li>An upload progress bar says the app is still working.</li>
<li>A disabled button says an action is not available yet.</li>
<li>A confirmation screen gives you a chance to catch a wrong bank account.</li>
<li>A badge shows that a notification is new.</li>
<li>A timestamp tells you whether the data is fresh.</li>
<li>A “try again” message tells you the failure may be temporary.</li>
</ul>

Good feedback is especially important when the action takes time. Without it, people click again, refresh repeatedly, or assume the app ignored them. Those extra actions can create duplicate orders, duplicate messages, or other confusing states.

### Why product thinking matters

Product thinking means understanding **the problem we are solving and who has it** before we decide what to build. The goal is not to collect feature requests or make a screen that sounds useful in the abstract. The goal is to reduce a real user pain in a real situation.

People often describe a solution because that is easier than describing the problem:

<div class="app-example">
<p><strong>Request:</strong> “Add a button that sends me a reminder every morning.”</p>
<p><strong>Possible problem:</strong> “I keep forgetting one important responsibility until it is too late.”</p>
</div>

Questions worth asking:

<ul class="question-list">
<li>Who is forgetting?</li>
<li>When does it happen?</li>
<li>What do they do today?</li>
<li>What would be good enough?</li>
<li>What is the smallest useful thing to try?</li>
</ul>

Traditional product discovery gives us a more complete picture:

<div class="app-example">
<p><strong>User:</strong> who experiences the problem?</p>
<p><strong>Situation:</strong> when and where does it happen?</p>
<p><strong>Current workaround:</strong> what do they do today instead?</p>
<p><strong>Pain:</strong> what is frustrating, costly, slow, risky, or repeatedly forgotten?</p>
<p><strong>Smallest useful intervention:</strong> what is the smallest change that could reduce that pain?</p>
<p><strong>Success signal:</strong> what would we observe if the pain actually became smaller?</p>
<p><strong>Not building yet:</strong> what attractive ideas are outside this first test?</p>
</div>

For the reminder example, “a morning reminder button” is only a proposed solution. The product questions are: whose responsibility is being forgotten, at what point does the failure matter, what reminder or coordination system exists today, and what specifically is painful about it? A successful first version might not be a reminder at all—it might make ownership visible, show one upcoming responsibility, or make it easy to confirm that a task was handled.

If we implement the request word for word, we might build a polished reminder button that does not help. Maybe the real problem is that the person does not know who owns the responsibility, or that the reminder arrives at the wrong time, or that there are too many reminders already.

This is why product judgment matters for developers. A developer who only translates user requests literally can produce exactly what was asked for and still miss the problem. Your job is not to reject requests or make every decision alone. Your job is to understand the situation well enough to ask a better question before building.

<aside class="model-purpose">
<p><strong>Product thinking in one sentence:</strong> Do not jump straight from “What should we build?” to code. First ask, “Whose pain are we reducing, in what situation, and what evidence would show that the pain actually became smaller?”</p>
</aside>

<ul class="example-list">

<li>The cart count stays unchanged after an item was added.</li>

<li>The app reports failure even though the action succeeded.</li>
