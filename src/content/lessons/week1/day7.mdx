## The complete story

Start with the user problem: a family needs one trustworthy chore list that different people can use.

<ol class="mental-model-steps">
<li>The user opens the client.</li>
<li>React displays the user-facing screen.</li>
<li>The client asks the server for chores.</li>
<li>The server applies the rules.</li>
<li>The server reads or changes rows in SQLite.</li>
<li>The server sends a response.</li>
<li>React updates the screen and gives honest feedback.</li>
<li>Another browser can request the same saved data.</li>
</ol>

Deployment is the missing connection between your computer and another person. It gives the client and server a reachable address. This lesson explains that idea, but does not teach a hosting provider, account setup, pricing, or provider-specific configuration.

## The important ideas from the whole week

The week introduced several technical ideas, but they all support one larger way of thinking. Keep these mental models available whenever you build or debug an app:

<table>
<thead><tr><th>Mental model</th><th>Question it helps you answer</th></tr></thead>
<tbody>
<tr><td><strong>Solve the problem, not the request</strong></td><td>Who is struggling, what are they trying to accomplish, and what is the smallest useful thing to try?</td></tr>
<tr><td><strong>User → data → action → rule → feedback</strong></td><td>What is the person doing, what information matters, what is allowed, and how will the app make the result clear?</td></tr>
<tr><td><strong>Temporary vs. saved data</strong></td><td>Should this information disappear after refresh, or should it be available later?</td></tr>
<tr><td><strong>React components and state</strong></td><td>Which part of the screen owns the current browser data, and which reusable pieces depend on it?</td></tr>
<tr><td><strong>Client → server → database</strong></td><td>Is the browser showing its own temporary copy, or is it displaying the shared data decided and saved elsewhere?</td></tr>
<tr><td><strong>Evidence and regressions</strong></td><td>What proves this change works, and what previously working behavior might have been broken?</td></tr>
</tbody>
</table>

### The course mantra

<aside class="mantra-callout">
<p>You are not here to create software.</p>
<p>You are here to solve user problems.</p>
</aside>

Users often describe the solution they imagine instead of the problem they are experiencing. Your job is to understand the person, situation, pain, and success signal before asking an agent to build anything.

### The five-part app loop

<ol class="mental-model-steps">
<li>A user sees or needs something.</li>
<li>The app has data representing the current situation.</li>
<li>The user takes an action.</li>
<li>A rule decides what is allowed.</li>
<li>The app gives feedback about what happened.</li>
</ol>

When debugging, walk through the same loop. Ask whether the user action happened, whether the rule ran, whether the data changed, and whether the screen told the truth.

### The layers of the Family Chore Board

<ol class="mental-model-steps">
<li>React components organize the user-facing client.</li>
<li>Browser state holds what this screen currently needs.</li>
<li>The client sends a request when shared data must be read or changed.</li>
<li>The server applies rules and decides what response to send.</li>
<li>SQLite stores the shared rows so they can return after refresh.</li>
<li>Verification checks whether each layer did its job.</li>
</ol>

The learner does not need to memorize every library or command. The important skill is being able to point to the layer responsible for a behavior and gather evidence there.


## Local, deployed, prototype, and production

<table>
<thead><tr><th>Term</th><th>Plain-English meaning</th></tr></thead>
<tbody>
<tr><td>Local</td><td>The client and server are running on your computer for development.</td></tr>
<tr><td>Deployed</td><td>The client and server are running somewhere reachable by a real URL.</td></tr>
<tr><td>Prototype</td><td>A small version that lets people try the important workflow.</td></tr>
<tr><td>Production</td><td>A system that has been hardened, supported, monitored, and prepared for real use.</td></tr>
</tbody>
</table>

Do not call a prototype production-ready just because it has a URL. A public URL does not prove that data is safe, persistent, or usable.

### Test the experience with a real person

If you already have a reachable URL, send it to one person. If you do not, run the client and server locally and let the person use the app while sitting with you. This lesson does not require learning a hosting provider before the real-user test.

Say:

<blockquote>
<p>Please try to add a chore, assign it, and mark it done. I will watch and take notes, but I will not explain every click.</p>
</blockquote>

Observe:

<ul class="question-list">
<li>What did they click first?</li>
<li>Where did they hesitate?</li>
<li>What did they expect to happen?</li>
<li>Did they notice waiting feedback?</li>
<li>Did they trust that the chore was saved?</li>
<li>Did they understand what to do next?</li>
</ul>

Then ask:

<ul class="question-list">
<li>What confused you?</li>
<li>What did you expect to happen?</li>
<li>Was any message unclear?</li>
<li>What would you change first?</li>
</ul>

Do not turn this into a feature wishlist. Separate an observed difficulty from a requested solution, then identify the smallest change worth considering.

### Ask Codex for the final test and edge-case audit

```text
Perform a critical final verification review.

1. Map every final acceptance criterion to a meaningful unit test,
   end-to-end test, or manual check.
2. Inspect available coverage and identify important code paths that are
   not covered.
3. Think through happy cases and edge cases, including blank input,
   repeated clicks, two chores with similar titles, a slow response, a
   failed response, refresh during a request, stale client data, and a
   second browser.

   Day 6.
```

### Populate the repository README

The repository’s `README.md` is the project’s front door. It is the first place a new person—or a future Codex agent—looks to understand what the project is, how to run it, how to check it, and what is still unfinished.

A useful README should answer practical questions without making someone reconstruct the project from scattered files:

<ul class="question-list">
<li>What problem does this project solve?</li>
<li>What can the current version do?</li>
<li>How do I start the client?</li>
<li>How do I start the server?</li>
<li>What does SQLite store?</li>
<li>How do I run the checks?</li>
<li>What environment settings are needed?</li>
<li>What is deliberately not built?</li>
<li>What is the current deployment status?</li>
<li>What limitations should I know about?</li>
</ul>

The README is documentation, not decoration. Writing it forces you to check whether the project can actually be explained and started by someone who did not build it. It also gives a future Codex agent high-level orientation before it reads individual files.

Do not let Codex make up commands, features, environment settings, deployment status, or security claims. The README should describe the repository that actually exists.

Use this prompt:

```text
Create or update the repository root README.md.

First inspect the actual repository. Read:
- docs/product-requirements.md, if it exists
- docs/high-level-design.md, if it exists
- docs/implementation-plan.md, if it exists
- package.json and all project scripts
- the React client entrypoints
- the server entrypoints
- the SQLite setup and schema creation code
- the available tests and verification commands

Do not change application code. Do not invent features, commands,
environment settings, deployment status, security guarantees, or test results.
If information is missing, say so clearly in the README.

Write a concise, beginner-friendly README with these sections:
1. Project name and one-sentence purpose
2. The user problem this project addresses
3. Current capabilities
4. How the client and server work together
5. What SQLite stores
6. Prerequisites
7. Setup instructions
8. How to start the client
9. How to start the server
10. How to run checks and tests
11. Important environment settings, listing names only and never values
12. Current deployment status
13. Known limitations and deliberately unbuilt features
14. Useful project structure notes

Use the exact commands found in the repository. Explain when two terminals
are required. Make clear that the client is user-facing, the server applies
rules and talks to SQLite, and SQLite is an existing open-source database
engine rather than code built by this project.

After writing the README:
- verify every command you document against package.json or the actual files
- run the documented check and build commands
- compare every capability claim with the implementation
- report anything you could not verify
- show a short summary of what the README now explains
```
