You finished Week 1 with a Family Chore Board that works on one computer. Week 2 asks a new question:

<aside class="model-purpose">
<p><strong>What has to change so another person can reach the same application through the internet?</strong></p>
</aside>

This is not a new feature sprint. The product stays the same. The goal is to move the jobs currently performed by your laptop onto services that other people can reach.

## Where you are starting

<pre class="astro-code"><code>One laptop
├── browser
├── React client
├── Node server
└── SQLite database file</code></pre>

The current board should still let someone view chores, add a chore, assign it, mark it done, reject invalid input, see useful feedback, refresh and recover saved chores, and open the same teaching data from another browser.

## Where you are going

By the end of Week 2, the system should tell this story:

<ol class="mental-model-steps">
<li>A person enters a public URL.</li>
<li>The browser finds the hosted application.</li>
<li>Vercel sends the client files to the browser.</li>
<li>The browser runs React.</li>
<li>React sends a request to a server route.</li>
<li>Vercel runs the server code for that request.</li>
<li>The server validates the request.</li>
<li>The server queries a Postgres database hosted by Supabase.</li>
<li>Supabase returns rows or an error.</li>
<li>The server sends a response.</li>
<li>React updates browser state and the screen.</li>
<li>The user receives accurate success or failure feedback.</li>
</ol>

The architecture is still one application because requests connect its parts.

## The names in the diagram

The diagram introduces three names that may be new. For now, think of them as hosted services taking over jobs your laptop currently performs:

<table>
<thead><tr><th>Name</th><th>Plain-language meaning</th><th>What it does for us in Week 2</th></tr></thead>
<tbody>
<tr><td><strong><a href="https://vercel.com/">Vercel</a></strong></td><td>A service that can build and make web application files reachable through a public URL.</td><td>Hosts the React client and, later, the server request code.</td></tr>
<tr><td><strong><a href="https://www.postgresql.org/">Postgres</a></strong></td><td>An open-source database that stores structured data in tables and rows.</td><td>A replacement for the SQLite database.</td></tr>
<tr><td><strong><a href="https://supabase.com/">Supabase</a></strong></td><td>A managed Postgres SaaS (software-as-a-service).</td><td>Hosts a Postgres database for us.</td></tr>
</tbody>
</table>

You do not need to memorize these names yet. The important question is: <strong>which job is each service taking over from the laptop?</strong>

<aside class="model-purpose">
<p><strong>SQLite and Postgres solve the same broad problem—storing structured data—but they fit different situations.</strong> SQLite keeps the database in a local file, which is convenient on one laptop. Postgres runs as a database service that a hosted server can reach, making it a better fit when the application and its shared data need to be available to people on different devices.</p>
</aside>

## What this week deliberately does not add

This week relocates the existing teaching application. It does not add:

<ul class="example-list">
<li>login, sessions, or password recovery;</li>
<li>family invitations, roles, or private per-user data;</li>
<li>browser-to-database access;</li>
<li>realtime subscriptions;</li>
<li>custom domains, analytics, payments, or file uploads;</li>
<li>background jobs, edge functions, or mobile apps; or</li>
<li>claims that this prototype is ready for private family information.</li>
</ul>

You do not need to memorize a long list of cloud vocabulary before starting. Each service will be introduced as an answer to a specific problem.

<table>
<thead><tr><th>Question</th><th>Answer for this week</th></tr></thead>
<tbody>
<tr><td>Where can the client files and server code run so another person can reach them?</td><td><a href="https://vercel.com/">Vercel</a>.</td></tr>
<tr><td>Where can shared data live so it does not depend on one local SQLite file?</td><td>A hosted Postgres database on <a href="https://supabase.com/">Supabase</a>.</td></tr>
</tbody>
</table>

The important thing is not to say “the cloud does it.” You should be able to name which part moved, where it moved, what request connects it, what data crosses the connection, and what evidence proves it worked.

## The working rhythm

Each day follows the same pattern:

<ol class="mental-model-steps">
<li>Understand the current system.</li>
<li>Define success.</li>
<li>Inspect what already exists.</li>
<li>Predict what should happen.</li>
<li>Move one responsibility.</li>
<li>Run focused checks.</li>
<li>Inspect the evidence.</li>
<li>Test the deployed environment.</li>
<li>Break one connection on purpose.</li>
<li>Explain what happened.</li>
<li>Save the work.</li>
</ol>

<aside class="mantra-callout">
<p>Do not begin with dashboards, accounts, keys, or deployment buttons.</p>
<p>Begin with the limitation you can already observe: the application works only while your laptop is acting as the whole world.</p>
</aside>
