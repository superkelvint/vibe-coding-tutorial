## The browser is public, so treat it as inspectable

Before we connect anything, make this rule permanent:

> Anything sent to the browser should be treated as public.

The page is not a private place. When someone opens the app, their browser downloads the JavaScript, HTML, and other files needed to display it. That person can open the browser’s developer tools, inspect the files, look at the requests, and copy anything that the page has received. A password does not become safe just because it is inside a JavaScript file, hidden behind a button, given an innocent-looking name, or made harder to read by a build tool.

This is why a database password, a private database connection string, or a powerful database key must never be placed in client code. If we put it there, every person who opens the app can potentially copy it. An attacker could then use that access outside the app: read information, change rows, delete data, or use the database for their own purposes. We should assume that anything the browser can receive can eventually be discovered. Putting a secret in the client is not a security measure; it is handing out the key.

The same rule applies to browser requests and responses. Do not return the database password in an API answer, print it in browser console output, put it in a visible error message, or paste it into a screenshot. A page may receive the chore rows it needs, but it should never receive the credential that lets it reach the database.

## What can go wrong if we ignore this rule?

If a secret or powerful database connection reaches the browser, the damage is not limited to someone seeing an ugly piece of code. A person who opens the app—or an automated attacker who examines it—may be able to:

<ul class="observation-checklist">
<li><strong>Steal the database password or private key.</strong> They can copy it and use it from their own computer.</li>
<li><strong>Read private rows.</strong> They may see information that the page was never supposed to show them, including personal, customer, employee, or company data.</li>
<li><strong>Change or delete rows.</strong> They may alter chores, remove records, or damage the database directly instead of using the page’s intended controls.</li>
<li><strong>Bypass the page’s rules.</strong> If the browser is the only place checking “is this allowed?”, an attacker can send a different request that skips those checks.</li>
<li><strong>Create fake or dangerous data.</strong> They may insert blank, enormous, misleading, or otherwise invalid values if the server does not validate them again.</li>
<li><strong>Impersonate another person.</strong> If a login token or similar credential is exposed, an attacker may act as the person who owns it.</li>
<li><strong>Use paid services at your expense.</strong> A stolen key can be used to send requests, consume storage, or create bills under your account.</li>
<li><strong>Discover internal details.</strong> Detailed database errors, private URLs, stack traces, and connection information can help an attacker plan the next attack.</li>
<li><strong>Copy or automate the requests.</strong> A person can replay, modify, or send thousands of requests even if the page only offers one small button.</li>
<li><strong>Trust a fake success message.</strong> If the browser is allowed to decide that an action succeeded, it can display success even when no authorized server operation occurred.</li>
</ul>

That is why the server must make the important decision again. The browser can ask, “Please add this chore,” but the server must decide whether the request is valid, whether the person is allowed to make the change, what fields may change, and whether the database confirms success. The browser is a useful user interface; it is not a trustworthy security boundary.

## The solution: let the server talk to Supabase

The server is the safer middle layer. In this lesson, that server code runs as a **Vercel Function**. “Safer” does not mean that the server is magically protected from attack. It means that the server runs in a different place from the browser, and that difference lets us keep private information and important decisions away from the user’s device.

Imagine a restaurant:

<ul class="question-list">
<li><strong>The menu is public.</strong> Anyone who visits can read it, copy it, or take a photograph of it.</li>
<li><strong>The customer can place an order.</strong> They can ask for a meal, but they do not get to walk into the kitchen and take ingredients.</li>
<li><strong>The kitchen staff checks the order.</strong> They decide whether it is a real menu item and whether it can be prepared.</li>
<li><strong>The pantry is locked.</strong> The customer does not receive the key to the pantry.</li>
<li><strong>The staff brings back the meal.</strong> The customer receives what they ordered, not the pantry key or every ingredient in the restaurant.</li>
</ul>

The app works the same way:

<table>
<thead><tr><th>Restaurant</th><th>App</th></tr></thead>
<tbody>
<tr><td>Public menu</td><td>React client: the page, buttons, and visible information sent to the browser.</td></tr>
<tr><td>Kitchen staff</td><td>Server / Vercel Function: checks the request and carries it out.</td></tr>
<tr><td>Locked pantry</td><td>Supabase database: shared saved information reached by the server.</td></tr>
<tr><td>Customer’s order</td><td>Browser request: “Please show these chores” or “Please add this chore.”</td></tr>
</tbody>
</table>

The React client is public because the browser needs it to display the page. The server is safer for private work because the browser does not receive the server’s private database key. The browser can ask for a result, just as a customer can place an order, but the server decides what to do and talks to the database on the browser’s behalf.

The important line is:

> Give the browser the page and the result it is allowed to see. Keep the database key with the server that uses it.

The restaurant analogy has one limit: someone can still send a fake order to the kitchen. That is why the server must check every request—titles, ids, permissions, and which fields may change—before it touches the database. Keeping the key out of the browser protects the key; it does not remove the need for the server to check its work.

<ol class="mental-model-steps">
<li>The browser sends a question to the server.</li>
<li>The server checks that the question makes sense.</li>
<li>The server uses private connection information to ask Supabase for rows.</li>
<li>Supabase reads or changes the durable row.</li>
<li>The server sends back only the answer the page needs.</li>
<li>The page updates what it displays.</li>
</ol>

This is safer because the private connection information stays with the Vercel Function. The browser sees only the request it is allowed to make and the limited answer the server chooses to return. The function can reject invalid input, decide which fields may change, and avoid exposing database details.

This is not magic protection. The server still needs careful validation, appropriate database permissions, useful error handling, and safe configuration. The important separation is that the browser is public while the database connection remains server-only.

The server needs a private way to contact Supabase. Supabase offers more than one connection option, so do not guess which one to use. Ask Codex to follow the current Supabase instructions for a Vercel Function and choose the appropriate server-only option. You do not need to memorize the option’s technical name.

<dl class="definition-list">
<div><dt>Private connection information</dt><dd>The address and credential the server needs to reach the database. It belongs on the server, not in browser code.</dd></div>
<div><dt>Server-only setting</dt><dd>A value that the Vercel Function can read but the browser must never receive.</dd></div>
<div><dt>Database query</dt><dd>A question or instruction sent to the database, such as “list the chores” or “save this new row.”</dd></div>
</dl>

## Give the server a private way to reach Supabase

The server, running as a Vercel Function, needs a private connection value so it knows which Supabase database to contact. This value must be available to the server, but it must never be sent to the browser.

You will learn environment settings in more detail on Day 13. For today, follow these concrete steps:

<ol class="mental-model-steps">
<li>Open your Supabase project and choose <strong>Connect</strong>.</li>
<li>Find the connection-string section and choose the option intended for a serverless function. If the dashboard shows several technical names, ask Codex which one the current Supabase instructions recommend for this project.</li>
<li>Copy the connection string. Treat it like a password. Do not paste the real value into this lesson, chat, notes, screenshots, or source code.</li>
<li>At the project’s top level, create or open a file named <code>.env.local</code>. This file is for private values used when the app runs on your laptop.</li>
<li>Put this line in <code>.env.local</code>, replacing only the placeholder with the real Supabase connection string:<br /><code>DATABASE_URL=PASTE_THE_SUPABASE_CONNECTION_STRING_HERE</code></li>
<li>Ask Codex to confirm that the server reads <code>process.env.DATABASE_URL</code> and that browser code does not read or import it.</li>
<li>In the Vercel project, open <strong>Settings → Environment Variables</strong>. Add a variable named <code>DATABASE_URL</code> and paste the real connection string into its private value field. Select the environments you are testing, such as Preview or Production.</li>
<li>Keep the actual value out of Git. This project’s <code>.gitignore</code> should ignore <code>.env</code> and <code>.env.*</code>; run <code>git status</code> and confirm that <code>.env.local</code> is not listed as a file to commit.</li>
<li>Never paste the connection string into a chat, screenshot, note, browser code, browser response, or committed file.</li>
<li>If the project has no safe place for this setting yet, stop and ask Codex to explain the smallest setup needed. Do not hard-code the value into application code.</li>
</ol>

For this lesson, the important variable is:

<table>
<thead><tr><th>Variable name</th><th>What it contains</th><th>Where it belongs</th></tr></thead>
<tbody>
<tr><td><code>DATABASE_URL</code></td><td>The private Supabase connection string.</td><td><code>.env.local</code> on your laptop and Vercel’s private environment settings online.</td></tr>
</tbody>
</table>

Do not put <code>DATABASE_URL</code> in a client-facing variable such as <code>VITE_DATABASE_URL</code>. The <code>VITE_</code> prefix means that a value is intended to be included in the browser build, which is exactly what we must avoid for a database connection.

You do not need to understand every part of the connection string today. The important rule is: the server can read it; the browser cannot. If you want to check the current recommendation, use <a href="https://supabase.com/docs/guides/database/connecting-to-postgres">Supabase’s connection documentation</a> or ask Codex to explain the relevant choice in plain English.

## Keep responsibilities in the right place

The page, server, and database each have a different job:

<table>
<thead><tr><th>Part</th><th>Job</th></tr></thead>
<tbody>
<tr><td>Browser page</td><td>Collect the person’s action, show loading or failure, and display the confirmed answer.</td></tr>
<tr><td>Server / Vercel Function</td><td>Check input, choose what may change, ask Supabase, and return a useful answer.</td></tr>
<tr><td>Supabase/Postgres</td><td>Store rows, apply table rules, create configured ids or times, and run queries.</td></tr>
</tbody>
</table>

The server should not blindly trust the browser. The browser should not receive the private database connection. The database should not be asked to invent product behavior that belongs in the server.

## Do not confuse “a Supabase table exists” with “the migration is complete”

At this point, it would be possible to have two databases:

<table>
<thead><tr><th>Old path</th><th>New path</th></tr></thead>
<tbody>
<tr><td>The app still reads and writes the local SQLite file.</td><td>A Supabase table exists, but the app does not use it yet.</td></tr>
<tr><td>The page shows local or sample data.</td><td>The page shows a row that someone inserted into Supabase by hand.</td></tr>
<tr><td>Changes work only on the laptop.</td><td>The hosted server reads and changes the shared Supabase rows.</td></tr>
</tbody>
</table>

Creating the Supabase project and table is preparation. The migration is complete for the hosted app only when the app’s real hosted paths use Supabase for every saved-data operation:

<ul class="observation-checklist">
<li><code>GET /api/chores</code> reads the current rows from Supabase.</li>
<li><code>POST /api/chores</code> creates the row in Supabase.</li>
<li><code>PATCH /api/chores/:id</code> changes the row in Supabase.</li>
<li>The browser receives the server’s answer, not a local SQLite result or a hard-coded sample list.</li>
<li>The hosted server does not open, import, or fall back to the laptop’s SQLite file.</li>
<li>A row added through the hosted app can be found in Supabase and returns after a refresh or from another browser.</li>
</ul>

The local SQLite database may remain as a Week 1 teaching artifact if the project still needs a clearly separate local version. That is not the same as the hosted app using SQLite. Document the separation: local-only SQLite is allowed; the hosted application’s saved-data path must use Supabase.
