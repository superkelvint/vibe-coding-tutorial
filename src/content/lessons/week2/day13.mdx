Yesterday, your online app learned how to ask Supabase for shared chores.

That created a new practical problem. The app can now run in several places:

<ul class="question-list">
<li>on your laptop while you are building it;</li>
<li>at a temporary preview address while you check a change; and</li>
<li>at the main address that other people use.</li>
</ul>

Those three versions run the same kind of code, but they still need to know where to find the database. They also need to keep the database connection private.

<aside class="model-purpose">
<p><strong>Today’s goal:</strong> learn how one app can use the right private settings in each place it runs.</p>
<p>You will make one small change, test it at a preview address, deliberately break only the preview connection, repair it, and confirm that production was not changed by the experiment.</p>
</aside>

<p><strong>What is intentionally out of scope:</strong> you are not adding a new product feature, creating a second real database, or building a complicated release system. You are learning one useful habit: make a small change, test it somewhere temporary, and protect the main version until you trust the result.</p>

## Start with the situation you can already see

Imagine you improve the loading message on the Family Chore Board. You want to check that improvement before changing the main online version.

You need two things at once:

<ol class="mental-model-steps">
<li>A way to try the change at a temporary online address.</li>
<li>A way to make sure that temporary version still uses the right database connection.</li>
</ol>

If the temporary version cannot find its database, the loading message is not the only thing you are testing. You may be looking at a configuration problem instead of a product change.

That is why we are learning this now. A preview is useful only if you understand what it is connected to and what it is safe to change there.

## What does “configuration” mean?

Configuration means information that tells an app about the place where it is running or the outside service it should use.

Think of a music app on your phone. The app’s buttons and rules are part of the app. The Wi-Fi password is not part of the app’s behavior; it is a setting that lets this phone reach one particular network.

The Family Chore Board has the same distinction:

<table>
<thead><tr><th>The app’s code</th><th>Its configuration</th></tr></thead>
<tbody>
<tr><td>Reject a blank chore title.</td><td>Which database the server should contact.</td></tr>
<tr><td>Show a saving message.</td><td>The private connection information for that database.</td></tr>
<tr><td>Ask for the current chores.</td><td>Whether this run is local, preview, or production.</td></tr>
<tr><td>Show an error when saving fails.</td><td>Other values that differ between your laptop and Vercel.</td></tr>
</tbody>
</table>

The code describes behavior. Configuration supplies values that may change depending on where that code runs.

<dl class="definition-list">
<div><dt>Code</dt><dd>The instructions that describe what the app does.</dd></div>
<div><dt>Configuration</dt><dd>Settings that tell those instructions which outside resource or environment to use.</dd></div>
<div><dt>Environment</dt><dd>One place where the app runs, such as your laptop, a temporary preview, or the main online version.</dd></div>
<div><dt>Preview</dt><dd>A temporary online version used to check a proposed change before it becomes the main version.</dd></div>
<div><dt>Production</dt><dd>The main online version intended for normal use.</dd></div>
</dl>

## One important limitation for this teaching project

For this beginner project, Preview and Production may use the same disposable Supabase database. That keeps the setup manageable, but it has a consequence: a chore you add while testing Preview may also appear in Production.

That is acceptable only because the chores are fictional teaching data. In a real application, a preview would usually need carefully separated data or carefully limited access so that testing could not change real users’ information.

<aside class="model-purpose">
<p><strong>Do not confuse “a different online address” with “a different database.”</strong></p>
<p>Preview and Production can be separate versions of the app while still pointing at the same saved rows. Check which arrangement this project is using before you test.</p>
</aside>

## Keep the database connection away from the browser

Yesterday you placed the private connection value in <code>.env.local</code> so the server could read it. Today, make the rule more precise:

<aside class="model-purpose">
<p><strong>The browser must never receive the database connection.</strong></p>
<p>The browser can receive the page, the chore rows it is allowed to display, and honest success or failure feedback. It must not receive the password or private connection information that lets the server reach Supabase.</p>
</aside>

The server reads the setting. The browser does not.

<table>
<thead><tr><th>Safe place</th><th>Unsafe place for <code>DATABASE_URL</code></th></tr></thead>
<tbody>
<tr><td><code>.env.local</code> used by the server on your laptop.</td><td>A React component or browser JavaScript file.</td></tr>
<tr><td>Vercel’s private environment settings.</td><td>A variable deliberately included in the browser build, such as <code>VITE_DATABASE_URL</code>.</td></tr>
<tr><td>Server code that reads <code>process.env.DATABASE_URL</code>.</td><td>A browser response, console message, screenshot, README, or Git commit.</td></tr>
</tbody>
</table>

<p>Remember: “server-only” describes who can read the value. It does not mean the value is magically harmless. Treat it like a password.</p>

## Set up the three environments

Use the following as a map. You do not need to memorize the dashboard language.

<ol class="mental-model-steps">
<li><strong>Local:</strong> use the project’s ignored <code>.env.local</code> file. It should contain the variable name <code>DATABASE_URL</code> and your real private value.</li>
<li><strong>Preview:</strong> in the Vercel project, open <strong>Settings → Environment Variables</strong>. Add <code>DATABASE_URL</code> for Preview and paste the private value into Vercel’s private field.</li>
<li><strong>Production:</strong> add the same variable name for Production using Vercel’s private field. Do not paste the value into the repository or into the client.</li>
</ol>

If you are unsure which Supabase connection option belongs in <code>DATABASE_URL</code>, ask Codex to check the current official Supabase guidance for this project. Do not paste the real value into the chat.

After adding or changing a Vercel setting, create a new deployment. A running deployment does not always pick up a setting that was changed after it was built.

Use this small configuration prompt if Codex finds that the project needs a fix:

```text
Make the smallest safe configuration fix for the Family Chore Board.

Required arrangement:
- the server/Vercel Function reads the server-only variable DATABASE_URL;
- local development reads it from the ignored .env.local file;
- Vercel Preview and Production can provide the same variable privately;
- no React/client/browser file reads or imports DATABASE_URL;
- no secret value is printed, returned, committed, or included in a build report.

Before editing, inspect the current project and explain the exact file that needs changing.
Do not redesign the app.
Do not add login, new database tables, or new product behavior.

After editing:
- run the existing checks and production build;
- confirm the browser bundle does not contain the real connection value;
- report file names and configuration names only, never secret values;
- explain what still needs to be configured in the Vercel dashboard by hand.
```

## Create and inspect the preview

Run the preview command Codex confirmed for this project. In the usual Vercel CLI workflow, <code>vercel</code> creates a preview deployment and <code>vercel --prod</code> is reserved for a production deployment. Use the project’s documented command if it differs.

When the command finishes, do not stop at “deployment succeeded.” Find the generated preview URL and open it in a fresh browser tab.

Check four things:

<ol class="mental-model-steps">
<li><strong>Find it:</strong> copy the preview URL from the terminal or Vercel dashboard.</li>
<li><strong>See the change:</strong> confirm the new loading message appears.</li>
<li><strong>Check the connection:</strong> open the browser Network panel and confirm the chore request reaches the expected online route and returns a successful response.</li>
<li><strong>Run regressions:</strong> load chores, add one valid chore, reject a blank title, mark one chore done, and refresh.</li>
</ol>

Compare the preview URL with the production URL. Production should still show the old loading message until you deliberately deploy the approved change.

## Deliberately break the preview connection

This is a safe way to learn what configuration is doing. Use only the disposable teaching project. Do not change production first, and do not use a real customer database.

Temporarily remove or misspell <code>DATABASE_URL</code> for Preview only. Do not delete the value from your local file or Production settings.

Predict before creating the broken preview:

<ul class="prediction-list">
<li>The page files may still load.</li>
<li>The server route may still be reachable.</li>
<li>The route that asks Supabase for chores should fail honestly.</li>
<li>Production should continue working because its setting was not changed.</li>
</ul>

Create a new preview deployment and inspect:

<ul class="observation-checklist">
<li>the page load;</li>
<li>the Network request and response status;</li>
<li>the visible error or failure feedback;</li>
<li>the Vercel function log, without exposing the secret value; and</li>
<li>the production page in a separate tab.</li>
</ul>

Then restore the Preview setting, create a new preview deployment, and prove that the chore request works again.

## If Preview works but Production fails

Preview and Production are separate deployments. A successful Preview does not automatically prove that Production has the same code, environment setting, database connection, or deployment timing.

If Preview works but Production fails, compare them in this order:

<ol class="mental-model-steps">
<li>Which commit is running in each deployment?</li>
<li>Does each environment have a variable named <code>DATABASE_URL</code>?</li>
<li>Was a new deployment created after the setting changed?</li>
<li>Does the browser send the same request path in both places?</li>
<li>Does the Vercel Function fail before contacting Supabase, while contacting it, or after receiving its answer?</li>
</ol>

Do not “fix” this by copying a secret into source code. Compare the setting’s name, environment, presence, and behavior without revealing its value.

## If a secret ever enters a tracked file

Do not use a real secret to practice this. You only need to understand the response.

If a real database password or connection string is ever committed to Git, deleting the line is not enough. The value may still exist in the project’s history, so stop and tell a mentor. The credential should be replaced, or “rotated,” so the old value no longer works.

For this lesson, the safe rule is simpler:

<ul class="question-list">
<li>Keep real values in <code>.env.local</code> and private Vercel settings.</li>
<li>Commit only the variable name or a fake example, never the real value.</li>
<li>Run <code>git status</code> and confirm that the local secret file is not listed.</li>
<li>Never paste the value into Codex, a screenshot, a README, or a browser response.</li>
</ul>

If you want to see what Git would notice, use a clearly fake value only:

<ol class="mental-model-steps">
<li>Create a temporary file such as <code>scratch-secret.txt</code> containing <code>FAKE_DATABASE_URL=not-a-real-secret</code>.</li>
<li>Run <code>git status</code> and observe that the file is visible as an untracked file.</li>
<li>Delete the temporary file without committing it.</li>
<li>Run <code>git status</code> again and confirm that it is gone.</li>
</ol>

This exercise demonstrates why <code>.env.local</code> is ignored. It does not make a real secret safe after it has been committed. If a real secret ever enters Git history, stop and ask a mentor to help replace it.
