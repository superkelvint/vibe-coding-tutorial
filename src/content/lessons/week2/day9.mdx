Today you will learn what Vercel is for, then deploy only the React client to it.

By the end, you will have a public URL that another person can open. The screen may appear there, while chore data still works only from your laptop. That is the expected result today—not a mistake.

## First, what problem is Vercel solving?

Right now, the Family Chore Board works because your laptop is doing several jobs: it runs the client, it runs the server, and it holds the SQLite file.

But what happens when you want another person to open the client while your laptop is closed?

Your laptop cannot be the public home of the application. We need another computer, already connected to the internet, to build and serve the client files.

<aside class="model-purpose">
<p><strong>Vercel is a service that can take a saved version of your project, build the client files, and make those files reachable through a public URL.</strong></p>
<p>Today Vercel takes over only the job of serving the React client. It does not automatically take over the Node server or the SQLite file.</p>
</aside>

### Git and Vercel have different jobs

These tools are not the same thing:

<table>
<thead><tr><th>Tool</th><th>Job</th></tr></thead>
<tbody>
<tr><td><strong>Git</strong></td><td>Keeps understandable save points in your project history.</td></tr>
<tr><td><strong>Vercel account</strong></td><td>Gives you a place to manage hosted deployments.</td></tr>
<tr><td><strong>Vercel CLI</strong></td><td>Lets you send the project from your terminal to Vercel and receive a deployment URL.</td></tr>
<tr><td><strong>Vercel</strong></td><td>Builds the client from the project and serves the result through a public URL.</td></tr>
</tbody>
</table>

You do not need to memorize every Vercel setting today. The important question is: <strong>which job is Vercel taking over from the laptop?</strong>

Day 9 uses the Vercel CLI to send the project directly from your computer. Vercel can also deploy from a GitHub repository: Git keeps the local history, GitHub stores an online project copy, and Vercel reads that copy to build a deployment. If you want to learn that workflow, use the standalone <a href={`${import.meta.env.BASE_URL}/lessons/git-and-github-guide/`}>Git and GitHub guide</a>. You do not need to switch deployment methods in the middle of this exercise.

The goal is not to make the whole application work from the public URL yet. The goal is to prove:

<aside class="model-purpose">
<p><strong>The client can be publicly reachable while the server and database are still local.</strong></p>
</aside>

Do not hide the incomplete part. Seeing exactly what did not move is the lesson.

Target architecture for this step:

- the current user interface
- existing client behavior
- existing API request shape
- existing local development workflow

Constraints:
- do not migrate the server yet
- do not migrate the database yet
- do not add authentication
- do not replace the framework
- do not hide failed requests

- run the production build

## Create a Vercel account

You need a Vercel account before you can create a hosted deployment. An account gives you a place to manage your deployments; it does not change the code in your project.

Open the official [Vercel sign-up page](https://vercel.com/signup) and create an account using an email address or a supported sign-in provider. Use an account you control and can access again later.

The CLI connects your local project folder to your Vercel account. It does not mean Vercel has moved your server or database.

You do not need a Vercel account to run the application locally. You need one for today’s hosted deployment step.

<Checkpoint id="week2-day9-vercel-account">I created or signed into a Vercel account and can explain that it manages deployments while the project source remains in my local project folder.</Checkpoint>

Install the Vercel command-line tool from your terminal:

```bash
npm install --global vercel
vercel --version
```

The first command installs the `vercel` command. The second confirms that your terminal can find it. Then connect the CLI to your account:

```bash
vercel login
```

Follow the instructions it gives you. Do not paste passwords, one-time codes, or secret values into Codex. Keep those private.

<Checkpoint id="week2-day9-vercel-cli">I installed the Vercel CLI, confirmed its version, and signed it into my Vercel account without sharing private login information.</Checkpoint>

## Open the deployed page and find the evidence

When the `vercel` command finishes successfully, it prints a public URL. It will look similar to:

```text
https://your-project-example.vercel.app
```

Copy that exact URL into a browser. Use a private or incognito window—a fresh browser window that does not reuse your normal page history—so you can see what a new visitor would see.

Check the page in this order:

<ol class="mental-model-steps">
<li>Does the URL open at all?</li>
<li>Does the page structure and styling appear?</li>
<li>Does the browser show the expected client files loading?</li>
<li>Does the page make an API request when it needs chores?</li>
<li>Does that request target <code>localhost</code> or a public host?</li>
<li>What status or error does the request show?</li>
</ol>

If you lose the URL, list recent deployments from the project folder:

```bash
vercel list
```

To inspect one public copy from the terminal, use its URL:

```bash
vercel inspect https://your-project-example.vercel.app --logs
```

You can also open the [Vercel dashboard](https://vercel.com/dashboard), choose the project, open **Deployments**, and select the public copy. The page contains the build result and build logs—text showing what Vercel did and where it stopped.

## Where to troubleshoot when something goes wrong

Start with the first place where the evidence differs from your prediction. Do not guess that the whole app is broken:

<table>
<thead><tr><th>What you see</th><th>Where to look first</th><th>What it may mean</th></tr></thead>
<tbody>
<tr><td>The <code>vercel</code> command ends with an error and no URL.</td><td>Read the terminal output from the command.</td><td>Vercel could not install the packages, prepare the files, or create the public copy.</td></tr>
<tr><td>The public copy exists but the page does not open.</td><td>Read the public-copy status and build logs in Vercel.</td><td>The build may have failed, the URL may be wrong, or access may be restricted.</td></tr>
<tr><td>The page opens but looks blank or has no styling.</td><td>Open the browser’s Network and Console panels. These show file requests and browser error messages.</td><td>The screen files may have loaded incompletely, or a file path may be wrong.</td></tr>
<tr><td>The page opens but the chore request fails.</td><td>Open the browser’s Network panel and copy the request URL, status, and error.</td><td>The client may still be asking <code>localhost</code>, which now means the visitor’s device.</td></tr>
<tr><td>A hosted server request later returns an error.</td><td>Read Vercel’s running-app logs and the request evidence in the browser.</td><td>The server code, its settings, or a later database connection may be failing.</td></tr>
</tbody>
</table>

Do not jump straight to “Vercel is broken.” First identify whether the problem happened while preparing the files, while opening the page, or when the browser asked for data.

## The important partial failure

At the public URL, a useful result is:

<ul class="observation-checklist">
<li>The screen loads from Vercel.</li>
<li>The request to a <code>localhost</code> API fails.</li>
</ul>

The browser is now running on the visitor’s device. When that browser requests <code>localhost</code>, it means the visitor’s device—not the developer’s laptop.

This is why the server must also be hosted.
