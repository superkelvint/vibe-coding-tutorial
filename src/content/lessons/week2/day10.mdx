That works for you while the program is running on your laptop. It cannot work for someone else’s browser, because their browser cannot reach a program running on your computer.

Today you will give that small program a place to run online, then prove that a browser can receive its answer.

The database is not part of today’s move.

## What problem are we solving today?

The problem we are solving today is simple:

> The page is online, but the program that supplies its chore data is still running only on your laptop.

Today, we will move that small program online too.

Today we will test the smallest useful step: can the page ask a small server program running online for an answer?

We will not connect the database today because that would make it hard to tell what went wrong. A problem could come from the page, the online program, the saved data, or the connection between them. One step at a time keeps the result understandable.

## One answer: put the program somewhere the page can contact it

The problem creates a natural question:

> If the page is online but the program is only on your laptop, where can the program run so that the page can contact it?

One answer is a **Vercel Function**. A Vercel Function is a small program that Vercel runs when a browser asks it a question. You do not keep the program running on your own laptop; Vercel starts it, lets it prepare an answer, and sends that answer back to the browser.

The program still does the same basic job:

<ol class="mental-model-steps">
<li>Receive a question from the page.</li>
<li>Read what the page asked for.</li>
<li>Apply the rules.</li>
<li>Use saved data later, when we connect a database.</li>
<li>Send an answer back.</li>
</ol>

Only the place and timing change:

<table>
<thead><tr><th>Program on your laptop</th><th>Vercel Function</th></tr></thead>
<tbody>
<tr><td>You start the program on your laptop.</td><td>Vercel runs the small program when a browser asks for an answer.</td></tr>
<tr><td>The program waits for questions while it is running.</td><td>Vercel starts the program when it is needed.</td></tr>
<tr><td>It can use the saved file on your laptop.</td><td>Today’s function returns a simple answer or temporary sample data.</td></tr>
</tbody>
</table>

We still have the same kind of server program. We are changing where and when it runs so the online page can contact it.

## See the smallest possible example

Before asking Codex to change the project, look at what a Vercel Function can be at its simplest. Imagine a file called <code>api/health.js</code>:

```js
export default function handler(request, response) {
  response.status(200).json({ status: "ok" });
}
```

That is the whole idea:

<ol class="mental-model-steps">
<li>The browser asks for <code>/api/health</code>.</li>
<li>Vercel runs this small function.</li>
<li><code>request</code> represents the browser’s question.</li>
<li><code>response</code> sends an answer back.</li>
<li><code>status(200)</code> says the answer succeeded.</li>
<li><code>{`json({ status: "ok" })`}</code> sends the answer in a format the page can read.</li>
</ol>

You do not need to memorize this code. The important idea is that a Vercel Function is not mysterious: it is a small piece of code that receives a question and sends back an answer. Codex will inspect this project and decide whether this exact shape, or the project’s existing equivalent, is the right way to add the health check.

<p><a href="https://vercel.com/docs/functions/runtimes/node-js">Vercel’s Node.js Function documentation</a> shows the same request-and-response idea in its official examples.</p>

## Ask Codex to move one small answer at a time

Do not ask Codex to move every server job in one request. Start with one tiny answer that can prove the online program is reachable.

Three small terms appear in the prompt below:

<dl class="definition-list">
<div><dt>GET</dt><dd>Ask a program to give you something.</dd></div>
<div><dt><code>/api/health</code></dt><dd>The address for this particular question. It is called a route.</dd></div>
<div><dt>JSON</dt><dd>A simple text format that lets a program send structured information to a browser.</dd></div>
</dl>

Use this prompt:

```text
I am a beginner moving one small server program onto Vercel.

Current situation:
The page is online, but the program that supplies its data still runs only on my laptop.

Goal:
Create one online GET /api/health route that returns JSON.

Expected response:
{ "status": "ok" }

Please:
- inspect the current repository before editing;
- explain where the program currently handles questions;
- explain how this project should run one Vercel Function;
- create only the health route first;
- preserve the local development workflow where practical;
- do not connect the saved data yet;
- do not add login or a new framework;
- do not move every existing route;
- inspect the existing checks and local commands before editing;
- add the smallest focused test only if the project already has a matching test pattern.

Acceptance criteria:
- GET /api/health returns a 2xx response;
- the JSON body is { "status": "ok" };
- the route works locally;
- the route works at the public Vercel URL;
- evidence can be inspected to show that the small program ran.

Report:
- exact files changed;
- exact commands run;
- local request and response;
- public request and response;
- what evidence proves the small program ran;
- what remains unverified.

Do not claim the database is connected.
```

Before sending it, predict what Codex should add and what it must leave alone.

## Tell the page where to ask for chores

Change the page so it asks for chores at this address:

```text
/api/chores
```

A path that starts with <code>/</code> means:

> Ask the same host that served this page, but use the <code>/api/chores</code> path.

Compare:

```text
http://localhost:3000/api/chores
```

with:

```text
/api/chores
```

Hard-coding <code>localhost</code> into the page asks the visitor’s own device. Starting with <code>/api/chores</code> asks the same online address that provided the page.

<img src={`${import.meta.env.BASE_URL}/images/week2-day10-localhost-trap.svg`} alt="Comparison showing localhost working from the developer’s laptop but failing from another visitor’s device, while a relative API path asks the public host." />
<p class="image-caption"><strong>What to notice:</strong> the word <code>localhost</code> changes meaning with the device. A relative path avoids asking a visitor to find a server on their own laptop.</p>

## Run every earlier check again

Adding an online answer can accidentally disturb something that already worked. A **regression** is an unintended break in an older behavior. Before you call Day 10 complete, rerun all of the relevant checks from Days 8 and 9:

<ol class="mental-model-steps">

<li>Confirm the page on your laptop can still get data from the program on your laptop.</li>
<li>Open the online page in a fresh browser window and confirm the page, styling, and images still load.</li>
<li>Confirm the online page does not pretend that saved data is connected.</li>
<li>Check the browser’s question-and-answer list for the requests you already understood from the earlier days.</li>
<li>Run the new laptop and online health checks, then compare the results with your prediction.</li>
</ol>
