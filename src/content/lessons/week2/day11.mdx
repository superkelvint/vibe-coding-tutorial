## Where should the chores live?

Imagine that your laptop is closed and another person opens the online page. The page still needs the same chore rows that you can see on your laptop.

Those rows need to survive after your laptop is closed and be available as one shared set of information.

Your laptop’s SQLite file was a good choice for the local version. It kept the data in one clear place next to the local program. The problem appears when the program runs online: there is no longer one dependable laptop file that every request should treat as the shared truth.

That means the program and the saved information now need separate homes:

<ol class="mental-model-steps">
<li>The page can be opened online.</li>
<li>The small server program can run online when the page asks it a question.</li>
<li>The shared chore rows need to remain available between questions.</li>
<li>The shared rows must not depend on your laptop being open.</li>
</ol>

## What does “shared saved information” mean?

Think about a spreadsheet:

<ul class="question-list">
<li>It can contain many rows.</li>
<li>Each column has a name.</li>
<li>Each row can describe one thing.</li>
<li>A later reader can see the same values instead of starting with a blank sheet.</li>
</ul>

A database table follows the same basic idea. For this course, the <code>chores</code> table will contain one row per chore, with columns for its title, assigned person, completion state, and creation time.

## One answer: use a separate hosted database

The problem creates a natural question:

> If the page and program can run online, where can the shared rows live so they do not depend on one laptop?

For this course, we will use **Supabase**. Supabase is a service that can run and manage a **Postgres database** for us.

You do not need to build Postgres yourself. Postgres is an open-source database program, like SQLite was in Week 1. We are using an existing implementation; our work today is to create the project and the table that matches the chore information we already have.

<dl class="definition-list">
<div><dt>SQLite</dt><dd>The database program used by the local version. Its saved information lives in a file on your laptop.</dd></div>
<div><dt>Postgres</dt><dd>Another database program. In this lesson, it will hold the shared rows in an online service.</dd></div>
<div><dt>Supabase</dt><dd>The service that gives us a hosted Postgres database and tools for inspecting it.</dd></div>
<div><dt>Table</dt><dd>A named collection of rows and columns, similar to a spreadsheet.</dd></div>
</dl>

The important continuity is:

<ol class="mental-model-steps">
<li>Both SQLite and Postgres hold tables.</li>
<li>Tables contain rows.</li>
<li>Rows contain named fields.</li>
<li>A program can ask the database to read or change rows.</li>
</ol>

Today we will create the hosted database and inspect it. Tomorrow the online program will learn how to ask it for chores.

## Keep a record of what exists

The dashboard lets you see the table that exists remotely. A migration file is a saved instruction in the project that records how to create that table again.

<dl class="definition-list">
<div><dt>Dashboard</dt><dd>A website where you can inspect and operate the hosted project.</dd></div>
<div><dt>Migration</dt><dd>A saved, reviewable instruction for creating or changing database structure.</dd></div>
</dl>

The dashboard answers, “What exists in this project right now?” The migration file answers, “What does this project expect the table to look like?” Keep both visible. A remote table should not become undocumented magic.

## Create the table: choose one of two methods

Both methods create the same table design you just compared with SQLite. Choose one method and complete it. Do not run both, because the second attempt will usually report that the table already exists.

### Method 1: Use the Visual Table Editor

This is the easiest option if you do not want to work with SQL. You fill in a form, like setting up columns in a spreadsheet.

<ol class="mental-model-steps">
<li>Open the project dashboard.</li>
<li>In the left-hand navigation, choose <strong>Table Editor</strong>. It may look like a small grid or spreadsheet icon.</li>
<li>Choose <strong>New table</strong>.</li>
<li>For the table name, enter <code>chores</code>. Use lowercase letters and underscores for database names.</li>
<li>Leave <strong>Enable Row Level Security (RLS)</strong> checked. This is a protection setting for data access; the application is not connecting today, so you do not need to create any access rules yet.</li>
<li>Add a column named <code>id</code>. Choose a whole-number type, make it the primary key, and enable automatic identity values if the form offers that option.</li>
<li>Add a column named <code>title</code> with type <code>text</code>. Mark it as required, usually shown as <strong>Is Nullable</strong> turned off.</li>
<li>Add a column named <code>assigned_to</code> with type <code>text</code>. Make it required too.</li>
<li>Add a column named <code>done</code> with type <code>boolean</code>. Set its default value to <code>false</code>.</li>
<li>Add a column named <code>created_at</code> with a date-and-time type. Set its default to the current time if the form offers a current-time option.</li>
<li>Choose <strong>Save</strong>. Wait for the table to appear in the Table Editor.</li>
</ol>

If the form uses slightly different labels, look for the same ideas: column name, type, required/not-null, primary key, identity/automatic number, and default value.

### Method 2: Use the SQL Editor

This method lets one reviewed instruction create the table and its rules. You do not need to write SQL from memory. ChatGPT or Codex can help produce it, and you review it before running it.

<ol class="mental-model-steps">
<li>Open the project dashboard.</li>
<li>In the left-hand navigation, choose <strong>SQL Editor</strong>. It may look like a terminal or code icon.</li>
<li>Choose <strong>New query</strong>.</li>
<li>Paste the reviewed table instruction below into the editor.</li>
<li>Read the table name and column names once more.</li>
<li>Choose <strong>Run</strong>, or use <kbd>Cmd</kbd> + <kbd>Enter</kbd> on Mac or <kbd>Ctrl</kbd> + <kbd>Enter</kbd> on Windows/Linux if that shortcut is shown.</li>
<li>Wait for a success message. If Supabase shows an error, stop and save the exact message before changing anything.</li>
</ol>

This is an example of the table instruction when the SQLite report contains the Week 1 fields shown here. Do not run it until the parity review says that it matches your real SQLite schema. If the report found different names or rules, use the corrected SQL from the parity review instead.

```sql
create table chores (
  id bigint generated by default as identity primary key,
  title text not null,
  assigned_to text not null,
  done boolean not null default false,
  created_at timestamptz not null default now()
);
```

## Break one database rule on purpose

Use the disposable teaching project or a safe, rolled-back experiment. Predict before each attempt.

### Failure 1: Use the wrong column name

In the SQL Editor, try this query:

```sql
select assignedTo
from chores;
```

The table has <code>assigned_to</code>, not <code>assignedTo</code>. Observe the exact database error, then do not save this query as your working query.

### Failure 2: Leave out a required value

Try this in the SQL Editor:

```sql
insert into chores (assigned_to, done)
values ('Taylor', false);
```

Because <code>title</code> is required, the database should reject the insert. Observe the exact error. No blank-title row should appear in the table.

### Failure 3: Ask for the wrong table

Try this:

```sql
select *
from chore;
```

The table is named <code>chores</code>, so the database should say that <code>chore</code> does not exist. Observe the exact error and compare the name with the table you created.
